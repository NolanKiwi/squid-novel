The Warehouse of Forgotten Things

Cache, Memory, and Forgetting

Everything changes after the second visit. The Storekeeper remembers what was already fetched and will not fetch it again. Remembering is the kingdom's strength; forgetting is its wisdom.

Chapter 1

The First Cache Hit

There is a moment that defines every caching proxy. The first time a requested object is already in storage. The first cache hit.

Before that moment, the proxy was merely a relay — a sophisticated middleman that added latency to every request. After that moment, it became something else: a warehouse that shortened the distance between desire and fulfillment.

The sequence was precise. The ACL judge had said ALLOW. ClientHttpRequest now consulted the Store::Controller. It called storeGetPublic() with the request's URL, passing through the key-generation machinery to produce a 128-bit MD5 hash. If that hash existed in the store index — if a StoreEntry was found with that key — the answer came not from the network but from memory or disk.

src/client_side_request.cc — the cache lookup moment
void
ClientHttpRequest::checkNoCache()
{
    if (r->flags.noCache) {
        // Client sent Cache-Control: no-cache — bypass the store
        doCallouts();
        return;
    }
    // Look up the store
    StoreEntry *entry = storeGetPublicByRequest(r);
    if (entry) {
        http->storeEntry(entry); // cache hit path
        processRequest();
    } else {
        // cache miss — must forward
        doCallouts();
    }
}

The first cache hit was the moment the warehouse justified its existence. Everything before it was infrastructure. Everything after it was reward.

Chapter 2

The Storekeeper

The warehouse had a master: Store::Controller, implemented in src/store/Controller.cc. It was the single authority over all stored objects — what was remembered, what was forgotten, how much space remained, which tier to use for each object.

The Controller's primary interface was simple: find(), create(), purge(). But behind these simple verbs was a system of extraordinary complexity — a three-tier hierarchy where each tier had different characteristics and different rules.

src/store/Controller.cc — the find() operation
StoreEntry *
Store::Controller::find(const cache_key *key)
{
    // Check transient store first (shared memory, SMP-safe)
    if (auto *e = transients->find(key))
        return e;

    // Check memory cache
    if (auto *e = mem_policy->findIn(memCache, key))
        return e;

    // Check disk cache
    for (auto *disk : swapDirs) {
        if (auto *e = disk->get(key))
            return e;
    }
    return nullptr;
}

The Controller was also the destination for new objects arriving from the network. When the Forward Commander returned with a response, the Controller received the response body byte by byte, decided whether it was cacheable, and if so routed it to the appropriate tier. All while the client was simultaneously reading from the same stream — a technique called streaming store-and-serve.

The Storekeeper never slept. Every byte that passed through the proxy was weighed, assessed, and either remembered or discarded.
Chapter 3

The Three Vaults

The warehouse was not a single room. It was a hierarchy of three distinct vaults, each with different speed, capacity, and purpose.

The first vault was Transients — shared memory in SMP mode, used to coordinate workers. When multiple workers received requests for the same uncached object simultaneously, the Transients vault prevented them from all forwarding at once. One worker became the fetcher; the others waited for the result.

The second vault was Memory Cache — RAM. Fast, expensive, limited. Objects here were served in microseconds. The memory cache held the hottest, most recently accessed objects. Its size was controlled by cache_mem in squid.conf.

The third vault was Disk Cache — the cache_dir hierarchy. Slow, cheap, large. Objects here were served in milliseconds. Squid supported multiple disk cache implementations: ufs (synchronous Unix I/O), aufs (async threads), diskd (separate disk daemon process), and rock (circular log-structured store).

Three-tier cache hierarchy
Transients (shared memory, ~MB)
    ↓ miss
Memory Cache (RAM, cache_mem, typically ~256MB)
    ↓ miss
Disk Cache (disk, cache_dir, typically ~GB-TB)
    ↓ miss
Network (origin server)
Speed and capacity are enemies. The vaults arranged them in truce: the fast was small, the large was slow, and between them a constant negotiation of what was worth keeping where.
Chapter 4

The Object's Life

Every cached object had a biography. It was born when a response arrived from the network and the Controller decided it was worth keeping. It lived in memory and disk, serving requests. And eventually it died — evicted by the replacement policy, expired by its headers, or purged by an administrator command.

The biography was encoded in a StoreEntry object. This structure carried the object's state through its entire lifecycle.

src/store.h — StoreEntry states
typedef enum {
    STORE_OK,           // fully fetched and stored
    STORE_PENDING,      // still being fetched from network
} store_status_t;

typedef enum {
    SWAPOUT_NONE,       // not written to disk
    SWAPOUT_WRITING,    // currently being written to disk
    SWAPOUT_DONE,       // fully written to disk
    SWAPOUT_FAILED,     // disk write failed
} swap_status_t;

While an object was in STORE_PENDING, it was simultaneously being written to disk and served to waiting clients. Multiple clients could attach to the same StoreEntry as StoreClients — they each maintained a read offset and were notified as new data arrived. The Forward Commander wrote to the front of the buffer; the StoreClients read from behind. The object grew while it was being consumed.

This was the streaming nature of the cache. No waiting for a full download before serving. No serving without also caching. One network connection feeding many clients and the disk simultaneously.

Chapter 5

The Key Maker

Every object in the warehouse had a key — a unique identifier used to locate it among millions of stored items. The key had to be fast to compute, compact to store, and collision-resistant enough that two different URLs would never accidentally map to the same key.

The solution was MD5. The Key Maker, implemented in src/store_key_md5.cc, computed a 128-bit MD5 hash of the URL (and optionally, for certain HTTP methods, the request body or headers).

src/store_key_md5.cc — key generation
const cache_key *
storeKeyPublicByRequest(HttpRequest * request)
{
    static cache_key digest[SQUID_MD5_DIGEST_LENGTH];
    // For most requests: hash of the URL
    // For VARY responses: hash of URL + Accept-Encoding etc.
    MD5Update(&M, (unsigned char *)url, strlen(url));
    MD5Final(digest, &M);
    return digest;
}

The 128-bit MD5 key was stored in the disk cache's index files and in the memory hash table. Lookup was O(1) — hash the URL, look up the key in a hash table, get the StoreEntry pointer.

MD5 was considered cryptographically broken by 2026 — but cache key generation did not require cryptographic security. It required speed and collision resistance for the URL space. In that role, MD5 remained perfectly adequate, and replacing it would have required migrating every existing cache file.

Some tools outlive the reasons they were chosen. MD5 became the cache's key not because it was best, but because it was first, and first is hard to displace.
Chapter 6

The Freshness Oracle

A warehouse full of stale goods serves no one. The kingdom needed an oracle to answer one question for every cached object: is this still fresh enough to serve, or must it be re-fetched from the origin?

The Freshness Oracle — src/refresh.cc — answered this question. Its primary function was refreshCheck(), which examined a StoreEntry and an incoming request and returned one of three verdicts: FRESH (serve from cache), STALE (must re-fetch), or STALE_BUT_USABLE (serve from cache but revalidate with origin).

src/refresh.cc — freshness determination hierarchy
// Priority order:
// 1. Cache-Control: max-age (if present in response)
// 2. Expires header (if present)
// 3. Last-Modified heuristic: age = (now - last_modified) * heuristic_pct
// 4. refresh_pattern rules from squid.conf

// Example:
// refresh_pattern -i \\.jpg$ 1440 20% 4320
//   min=1440min, pct=20%, max=4320min
//   if LM-heuristic-age is in [min, max], use it
//   else if outside range, clamp to range

The LM-Factor heuristic was based on a simple observation: documents that haven't changed in a long time are likely to continue not changing. If a document's Last-Modified header showed it was last modified 100 days ago and it had been in cache for 10 days, it was probably still fresh. The default heuristic used 10% of the age since last modification as the estimated freshness lifetime.

The refresh_pattern directive gave administrators fine-grained control over these defaults — different rules for images, scripts, HTML pages, API responses.

Chapter 7

The Pattern of Patterns

The refresh_pattern directive was a list of rules, each consisting of a regex pattern, a minimum freshness time, a percentage heuristic, and a maximum freshness time. The Freshness Oracle walked this list and applied the first matching rule.

Common configurations were well-understood recipes. Static assets — JavaScript, CSS, images with content-hash filenames — could be cached for months or years with max set high. Dynamic API responses needed short freshness windows. Error pages should not be cached at all.

typical refresh_pattern configuration
# Static assets with hash in filename: cache indefinitely
refresh_pattern -i \.[0-9a-f]{8}\.(js|css|woff2)$ 525960 100% 525960

# Images: allow up to 7 days
refresh_pattern -i \.(jpg|jpeg|png|gif|webp)$ 1440 20% 10080

# HTML: conservative, 1 hour max
refresh_pattern -i \.html?$ 0 10% 60

# Default for everything else
refresh_pattern . 0 20% 4320

The Pattern of Patterns was the administrator's primary tool for tuning cache behavior. A well-configured refresh_pattern could dramatically improve hit ratios; a poorly configured one could cause stale content to be served for days or force unnecessary re-fetches of objects that hadn't changed.

Every rule in the pattern was a bet: a bet about the future behavior of the content it matched. The administrator placed these bets with incomplete information, and the cache paid out accordingly.
Chapter 8

The Forgetfulness Law

The warehouse was finite. Disks filled up. RAM was limited. At some point the Storekeeper had to choose: what to discard to make room for the new? This was the domain of the replacement policy — the law of forgetting.

The replacement policy was defined by the cache_replacement_policy directive in squid.conf, and implemented in the src/repl/ directory. Two policies shipped with Squid: LRU (Least Recently Used) and LFUDA (Least Frequently Used with Dynamic Aging).

The policy was consulted whenever the Storekeeper needed to evict objects to make room. The policy maintained its own data structure alongside the store — a structure that could efficiently identify the "least valuable" object according to its specific metric.

src/repl/repl_modules.cc — policy interface
// Each replacement policy implements this interface:
struct _RemovalPolicy {
    void (*Add)(RemovalPolicy *, StoreEntry *, RemovalPolicyNode *);
    void (*Remove)(RemovalPolicy *, StoreEntry *, RemovalPolicyNode *);
    void (*Referenced)(RemovalPolicy *, StoreEntry *, RemovalPolicyNode *);
    RemovalPolicyWalker *(*WalkInit)(RemovalPolicy *);
};

The policy received notifications: Add (a new object entered the store), Remove (an object was evicted), Referenced (an object was accessed). It used these notifications to maintain its ordering structure. When eviction was needed, it walked the structure in eviction order and removed objects until enough space was freed.

Chapter 9

The LRU Wheel

LRU was the simplest and most intuitive replacement policy: evict the object that was least recently accessed. The assumption was that recently accessed objects were likely to be accessed again soon, and rarely accessed objects were unlikely to be needed.

The implementation was a doubly-linked list. Every time an object was referenced, it was moved to the front of the list. The back of the list held the least recently used objects. Eviction always removed from the back.

src/repl/lru/lru.cc — O(1) operations
static void
lru_Referenced(RemovalPolicy * policy, StoreEntry * entry,
               RemovalPolicyNode * node)
{
    LruPolicyData *lru = (LruPolicyData *) policy->_data;
    LruNode *lnode = (LruNode *) node->_data;
    // Move to front: O(1) linked list splice
    dlinkDelete(&lnode->node, &lru->list);
    dlinkAdd(entry, &lnode->node, &lru->list);
}

Every LRU operation — insert, remove, reference — was O(1). The doubly-linked list enabled constant-time splice operations: unhook a node from its current position, relink the neighbors, insert the node at the front. Three pointer updates. No traversal, no sorting, no rebalancing.

LRU was excellent for workloads with strong temporal locality — where recently accessed content was likely to be accessed again. It struggled with scan patterns, where a large sequential scan would evict all hot objects to make room for cold ones.

The LRU Wheel turned smoothly because it never tried to be clever. It remembered only one thing: when you were last needed.
Chapter 10

The Frequency Counter

LFUDA — Least Frequently Used with Dynamic Aging — was LRU's more sophisticated sibling. Instead of tracking recency, it tracked frequency: how many times had each object been requested? The least-requested object was evicted first.

But pure frequency had a problem: objects cached long ago might have accumulated high frequency counts that could never be overtaken by newer objects, even if the newer ones were more popular now. To compensate, LFUDA used dynamic aging: a global age counter that was added to newly inserted objects, effectively reducing the relative value of old high-frequency counts.

src/repl/lfuda/lfuda.cc — heap-based eviction
// Key formula:
// eviction_key = refcount + age_factor
// where age_factor is the global "cache age" counter

// When evicting: remove the object with the lowest key
// This is a min-heap operation: O(log N)
static void
lfuda_Referenced(RemovalPolicy * policy, StoreEntry * entry, ...)
{
    LfudaNode *node = ...;
    node->key = ++node->refcount + ldata->cache_age;
    heap_update(ldata->theHeap, node->heapNode, ...);
}

LFUDA used a min-heap. Inserting or updating a node was O(log N). Finding the minimum (the eviction candidate) was O(1). For large caches with millions of objects, this was slightly more expensive than LRU's O(1) operations — but LFUDA provided much better hit rates for workloads with stable popularity distributions, like large media repositories.

The choice between LRU and LFUDA was one of the cache tuning questions that distinguished novice administrators from experienced ones.

Chapter 11

The Disk Scribes

Memory was temporary. When Squid restarted, the memory cache vanished. The disk cache survived reboots, survived process crashes, accumulated years of stored content. The disk was the true long-term memory of the warehouse.

Writing to disk was a delicate art. Synchronous writes blocked the event loop — an intolerable condition in a single-threaded proxy. Squid offered three strategies for avoiding this problem.

The first was aufs — async UFS, which used a thread pool for all disk I/O. The event loop dispatched write requests to threads; the threads performed the actual syscalls; completion was reported via a notification pipe that the event loop could select() on.

The second was diskd — a separate process dedicated to disk I/O, communicating with the main process via IPC message queues. Even more isolated than threads, but with higher communication overhead.

The third was rock — a log-structured store that serialized all writes into a single sequential append-only file. Sequential writes are faster than random writes, and a circular log structure enabled efficient space reuse.

cache_dir type comparison
cache_dir ufs  /var/spool/squid 10000 16 256  # classic, sync
cache_dir aufs /var/spool/squid 10000 16 256  # async threads
cache_dir rock /var/spool/squid 10000 # circular log, SMP-friendly
The disk scribes worked in shifts that the main process never saw. They were the kingdom's night workers, inscribing memories onto platters while the Gatekeeper slept between requests.
Chapter 12

Collapsed Forwarding

In a busy proxy, the same uncached object might be requested by hundreds of clients simultaneously — a thundering herd. Without coordination, each request would trigger a separate forward to the origin server, multiplying load by hundreds.

Collapsed Forwarding prevented this. When the first request for an object triggered a forward, subsequent requests for the same object were collapsed — attached to the same StoreEntry as additional StoreClients. They would all receive the response as it streamed in, from a single origin request.

src/CollapsedForwarding.cc — the thundering herd solution
// Worker A misses on /heavy-file.iso, starts fetching
// Worker B also misses, but:
bool
CollapsedForwarding::Collapse(StoreEntry *e)
{
    // Atomically check if another worker is already fetching
    if (e->hittingRequiresCollapsing()) {
        // Yes! Attach to the existing entry as a reader
        e->lock("collapsed");
        return true; // don't forward again
    }
    return false; // we're the fetcher
}

In SMP mode, Collapsed Forwarding required inter-worker coordination through shared memory — the Transients store. Worker A wrote the response to the Transients store; Workers B, C, D read from it. The Coordinator process mediated the handshake.

Collapsed Forwarding was one of the features that made a heavily-loaded Squid dramatically more efficient than a collection of independent proxies. The many became one, and the one fetched for all.

Chapter 13

The Vary Puzzle

HTTP had a complication that made simple URL-keyed caching insufficient: the Vary header. When a server responded with Vary: Accept-Encoding, it meant "this response is specific to the Accept-Encoding header you sent. A client with a different Accept-Encoding might get a different response."

This meant that the same URL might have multiple valid cached representations — one for clients that accepted gzip, one for those that didn't. The cache key could not be just the URL; it had to include the values of the Vary headers.

src/HttpReply.cc — Vary handling
// When storing a response with Vary:
// 1. Store the "Vary key" as a secondary key on the StoreEntry
// 2. The vary key = MD5(URL + sorted-vary-header-values)

// When looking up a request:
// 1. Find entry by primary key (URL only)
// 2. If found, check if vary key matches
// 3. If vary keys differ, it's a miss — the stored version
//    was for different headers

The Vary puzzle was one of the harder problems in HTTP caching. Multiple variants of the same URL could exist in the cache simultaneously, each with a different vary key. The cache had to store all of them, find the right one for each request, and evict them independently.

The Vary Puzzle had no elegant solution — only careful bookkeeping.

Chapter 14

The Purge Command

Not all forgetting was governed by replacement policies or freshness rules. Sometimes an administrator needed to immediately remove a specific object from the cache — because it contained incorrect content, sensitive information, or had been updated at the origin and the cache needed to be cleared before its natural expiration.

This was the PURGE method — an HTTP method not defined in the standard but implemented by Squid as an administrative command. An HTTP PURGE request to the proxy with a specific URL would cause the Storekeeper to remove all cached representations of that URL.

PURGE request example
PURGE http://example.com/article/12345 HTTP/1.1
Host: example.com

# Squid response:
HTTP/1.1 200 OK
X-Cache: PURGED

# Or if not found:
HTTP/1.1 404 Not Found

The PURGE command was available only to IP addresses listed in the purge ACL. Without this protection, any client could evict any cached content. With it, only trusted management systems could issue purge commands — content delivery networks, CMS platforms, deployment systems that needed to invalidate cached content after updates.

The power to forget was as important as the power to remember. Without purging, the warehouse accumulated errors alongside truths, and could not be corrected without being demolished.
Chapter 15

Memory's Purpose

Why did the warehouse exist? The answer seemed obvious: to save bandwidth. To reduce load on origin servers. To speed up responses for end users. All of these were true.

But the deeper answer was resilience. A proxy with a large cache could serve clients even when origin servers were slow, overloaded, or temporarily unavailable. The warehouse was not merely an optimization — it was a buffer between the chaos of the open internet and the stability expected by users.

In the early days of the internet, when bandwidth was measured in kilobits and origin servers routinely fell over under modest load, caching proxies were essential infrastructure. Every university, every ISP, every large organization ran a Squid proxy — or something like it — to make the internet usable.

The cache hit ratio — the warehouse's report card
# cachemgr.cgi output:
Hits as % of all requests:     67.3%
Hits as % of bytes sent:       71.2%
Memory hits:                   34.1%
Disk hits:                     33.2%
Misses:                        32.7%

# A hit ratio of 67% means 2 out of 3 requests
# never touched the network.

In modern networks — with CDNs, HTTP/2, and gigabit connections — the case for forward proxies was less urgent. But Squid persisted. In corporate environments for content filtering. In developing regions with expensive bandwidth. In content delivery networks as a reverse proxy. The warehouse found new purposes as old ones faded.

The Storekeeper adapted. The warehouse endured.

Memory is not just about the past. It is about making the future cheaper to reach.
