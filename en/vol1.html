The Gates Open

Birth of the Proxy Kingdom

Everything begins with a single command.
The machine opens its eyes, reads the rules, starts its heartbeat,
and throws open the gates to greet the first visitor.

Chapter 1 The Primordial Silence

Three in the morning. The machine room was cold.

Row upon row of server racks stretched into the dark. Blue LED indicators blinked at irregular intervals. Cooling fans ran in a low, continuous hum — the breathing of something that never slept. And in the very center of it all, a single server sat waiting for its moment.

Its name was printed on a label stuck to the chassis: SQUID PROXY — v6.x.

The server had been powered down. Completely silent. No processes, no threads, no file descriptors open. Pure hardware waiting for electricity and intention.

Then someone typed a command.

$ squid -f /etc/squid/squid.conf

The shell passed the string to the kernel. The kernel loaded the binary from disk into memory. Text segment, data segment, stack, heap — all arranged precisely. And then it called main().

The kingdom was about to be born.

▼ src/main.cc — The very first lines
int
main(int argc, char **argv)
{
    return SquidMainSafe(argc, argv);
}

int
SquidMainSafe(int argc, char **argv)
{
    try {
        return SquidMain(argc, argv);
    } catch (const std::exception &e) {
        std::cerr << "FATAL: " << e.what() << std::endl;
        return 1;
    } catch (...) {
        std::cerr << "FATAL: unknown exception" << std::endl;
        return 1;
    }
}

The outermost shell. SquidMainSafe() wraps everything in a try-catch — one last line of defence before the process crashes. SquidMain() is where the real work begins.

Inside SquidMain() lay the roadmap of a kingdom's construction. Modules loaded. Command-line arguments parsed. Memory systems initialized. Config file found. Workers started.

One line at a time, the kingdom assembled itself.

⁂

The first step was registering all the modules. RegisterModules().

Squid was not a monolith. It was a federation. Dozens of modules — DNS, cache, ACL, SSL, ICAP, logging — each needed to announce itself to the central registry before it could be used. Like nobles swearing fealty before a coronation.

void
RegisterModules()
{
    Acl::RegisterMaker("src",    &AclMakerWrapper<ACLIP>);
    Acl::RegisterMaker("dst",    &AclMakerWrapper<ACLIP>);
    Acl::RegisterMaker("time",   &AclMakerWrapper<ACLTime>);
    Acl::RegisterMaker("port",   &AclMakerWrapper<ACLIntRange>);
    Acl::RegisterMaker("method", &AclMakerWrapper<ACLMethod>);
    // ...dozens more
}

Each RegisterMaker() call stored a factory function in a global map. Later, when the config parser encountered acl mynet src 192.168.0.0/16, it would look up "src" in the registry and call its factory to produce the right ACL object. The registry was the kingdom's phonebook.

Chapter 2 The Birth of main()

Memory had to be prepared before any serious work could begin. Mem::Init().

Squid was pathologically obsessed with memory. It had its own pool allocator, its own string type, its own reference-counting framework. All of it had to be bootstrapped in the right order, because each piece depended on the pieces below it.

▼ src/mem/Pool.cc — Memory pool initialization
void
Mem::Init(void)
{
    // Register all the standard-size pools
    // Pools are indexed by object size
    // e.g., pool for 64-byte objects, 128-byte objects, etc.

    mempool_init();

    // Register pools for known types
    // This is where MEMPROXY_CLASS classes get their pools
    mem_init_mallocs();
}

Squid's memory pool system pre-allocates blocks of memory and hands them out without calling malloc() each time. When an object is "freed," it goes back into the pool rather than being returned to the OS. This dramatically reduces malloc/free overhead at high request rates.

With memory ready, the filesystem module initialized, and ACL factories registered, it was time for the most important step: reading the config file.

The config file was the kingdom's constitution. squid.conf. Everything — cache size, access rules, listening ports, peer caches, logging format — lived in that single text file. Without it, the kingdom had no shape, no purpose, no identity.

The Law Scribe was summoned.

Chapter 3 The Law Scribe's Mission

The Law Scribe was ConfigParser. Its job: read the squid.conf file, one line at a time, and build the kingdom's internal model of the rules.

▼ src/ConfigParser.cc — Line-by-line parsing
void
ConfigParser::ParseFile(const char *filename)
{
    FILE *fp = fopen(filename, "r");
    if (!fp) {
        fatalf("Cannot open config file: %s", filename);
    }

    char line[BUFSIZ];
    while (fgets(line, sizeof(line), fp)) {
        // Strip comments (# to end of line)
        // Strip leading/trailing whitespace
        // Handle line continuations (\)
        // Skip blank lines

        // Tokenize and dispatch to handler
        parseConfigFileLine();
    }
    fclose(fp);
}

// The dispatch table — generated from cf.data.pre
// Each directive name maps to a handler function
// e.g., "http_port" → parse_http_port()
//       "cache_dir" → parse_cache_dir()
//       "acl"       → aclParseAclLine()

The config parser is largely auto-generated from cf.data.pre, a meta-description of every config directive. Running make regenerates cf_parser.cci — a giant dispatch table mapping directive names to handler functions. This way, adding a new directive only requires editing one place.

The parser moved through the file methodically. Each non-blank, non-comment line was a directive. The first word on the line was the directive name. The rest were its arguments.

When the parser hit acl, it called aclParseAclLine(). That function read the ACL name, looked up the type in the registry, called the factory, and stored the resulting Acl::Node in a global map. Later, http_access allow mynet would retrieve that ACL by name and wire it into the access control tree.

The constitution was being written, article by article.

Chapter 4 Contents of the Scroll

What was actually in a typical squid.conf? Here is a minimal but real configuration.

http_port 3128

cache_dir ufs /var/spool/squid 10240 16 256
cache_mem 256 MB
maximum_object_size 100 MB

acl localnet src 192.168.0.0/16
acl SSL_ports port 443
acl Safe_ports port 80 443 8080 21 70 210 1025-65535
acl CONNECT method CONNECT

http_access deny !Safe_ports
http_access deny CONNECT !SSL_ports
http_access allow localnet
http_access deny all

cache_mgr admin@example.com
access_log /var/log/squid/access.log squid

Each line was a story.

http_port 3128 told the kingdom where to hang its front door. Port 3128, the traditional proxy port. Clients would connect here.

cache_dir ufs /var/spool/squid 10240 16 256 established the stone warehouse — 10 gigabytes on disk, laid out in a 16×256 two-level directory hierarchy. That's 4,096 subdirectories to spread the load.

http_access deny !Safe_ports was the first of the Judge's laws. "Deny anyone trying to reach a port that isn't on the safe list." Stop port-scanning abuse at the gate.

cf.data.pre is the single source of truth for all config directives. The Perl script cf_gen.pl reads it and generates cf_parser.cci — a massive C switch/table. This is a classic code-generation pattern; hand-writing handlers for 200 directives would be error-prone and tedious.

Chapter 5 The Retainers Awaken

Config was loaded. The kingdom's shape was defined. Now the retainers had to be roused.

mainInitialize() was the great awakening ceremony. Each subsystem's init() or Init() function was called in careful order, because the order mattered — you can't open network sockets before the event system is ready; you can't start the cache before the filesystem is checked.

static void
mainInitialize(void)
{
    ipcache_init();           // DNS resolver cache
    fqdncache_init();         // Reverse DNS cache
    Dns::Init();              // Async DNS subsystem

    storeInit();              // Cache: open/rebuild swap dirs
    storeFsInit();            // Filesystem drivers

    neighbors_init();         // Load cache peer list
    neighborsRegisterWithCacheManager();

    clientdbInit();           // Per-client stats table

    // Periodic timers
    eventAdd("storeMaintain", storeMaintain, nullptr, 1.0, 1);
    eventAdd("ipcache_purgelru", ipcache_purgelru, nullptr, 10.0, 1);
    eventAdd("fqdncache_purgelru", fqdncache_purgelru, nullptr, 15.0, 1);

    serverConnectionsOpen();  // Bind and listen on http_port
    // ...
}

Each eventAdd() planted a recurring timer. storeMaintain would fire every second to prune expired cache entries. ipcache_purgelru would fire every 10 seconds to evict old DNS entries. These timers were the kingdom's heartbeat — the rhythmic maintenance that kept the system healthy over time.

Then serverConnectionsOpen(). The moment of truth. Squid bound its listening socket to port 3128 and called listen(). The front door was now physically open. Clients could knock. But nothing would answer yet — the event loop hadn't started.

Chapter 6 The Great Wheel's Heart

Everything that had happened so far — module registration, config parsing, memory initialization, subsystem startup — was just preparation. It was all synchronous, all sequential, all blocking. One thing after another in a straight line.

Now came the thing that made Squid Squid.

mainLoop.run(). The event loop. The Great Wheel.

▼ src/EventLoop.cc — The infinite loop
void
EventLoop::run()
{
    last_loop = false;
    while (!last_loop) {
        runOnce();
    }
}

bool
EventLoop::runOnce()
{
    bool sawEvents = false;

    // 1. Run all registered engine callbacks
    //    (timers that have fired, signals, etc.)
    for (auto &engine : engines) {
        int timeout = engine->checkEvents(0);
        if (timeout == AsyncEngine::EVENT_IDLE) continue;
        sawEvents = true;
    }

    // 2. Flush the async callback queue
    while (AsyncCallQueue::Instance().fire()) {
        sawEvents = true;
    }

    // 3. Wait for I/O events (epoll_wait / kevent)
    //    Timeout: until the next timer fires
    comm_select(calculateTimeout());

    return sawEvents;
}

The event loop is Squid's soul. It runs forever — or until shutdown is signaled. Each iteration: fire pending callbacks, drain the async queue, then block on epoll/kqueue until something happens (socket readable, timer expired, signal received). That "block on epoll" is where the CPU actually rests.

The Great Wheel began to turn. while(!last_loop). It would turn until someone killed the process or sent a shutdown signal. Minutes, hours, days, years — it didn't matter. It just kept turning.

In between turns, everything happened. DNS replies arrived. New TCP connections knocked. Cache entries expired. Peer caches sent ICP hits. Access log buffers were flushed. Each of these was an event, and each event triggered callbacks that had been registered earlier.

The magic: none of this blocked. A single OS thread handled it all. While waiting for a DNS reply, it served five other requests. While writing a cache file to disk, it parsed three incoming HTTP headers. Total concurrency from zero threads.

Chapter 7 Secrets of the Pulse

What actually happened inside comm_select() — the wait-for-I/O step?

On Linux, it was epoll_wait(). On BSDs and macOS, kevent(). On ancient systems, select() or poll(). All of them did the same thing: hand the OS a list of file descriptors to watch and a timeout. The OS put the process to sleep. When any of those file descriptors became readable or writable — or when the timeout expired — the OS woke the process up and reported what happened.

▼ src/comm/Select.cc — epoll inner loop
int
Comm::DoSelect(int msec)
{
    struct epoll_event events[MAX_POLL_TIME];
    int num = epoll_wait(kdpfd, events, MAX_POLL_TIME, msec);

    if (num < 0) {
        if (errno == EINTR) return COMM_OK; // signal interrupted
        return COMM_ERROR;
    }

    // For each ready file descriptor...
    for (int i = 0; i < num; ++i) {
        int fd = events[i].data.fd;
        bool read_ready  = events[i].events & EPOLLIN;
        bool write_ready = events[i].events & EPOLLOUT;

        // Schedule the registered callbacks
        if (read_ready  && fd_table[fd].read_handler)
            ScheduleCallHere(fd_table[fd].read_handler);
        if (write_ready && fd_table[fd].write_handler)
            ScheduleCallHere(fd_table[fd].write_handler);
    }

    return COMM_OK;
}

epoll_wait() returns when one or more sockets are ready. Squid doesn't immediately call the handlers — it schedules them via ScheduleCallHere() onto the AsyncCallQueue. The queue is then drained in the next phase of runOnce(). This two-phase approach keeps the event loop's internal state consistent.

Time was a first-class citizen in this world. squid_curtime — a global variable holding the current Unix timestamp — was updated every loop iteration. Most of Squid didn't call gettimeofday() directly; instead, they read squid_curtime. It was a small optimization with a big impact: at 10,000 requests per second, calling gettimeofday() 10,000 times per second added up fast.

Chapter 8 The Gates Open

The event loop was running. A file descriptor — the listening socket — was registered with epoll. And someone knocked.

A client machine somewhere on the network initiated a TCP connection to port 3128. The kernel completed the three-way handshake without waking Squid. Then epoll noticed the listening socket had become readable — meaning a completed connection was waiting in the accept queue — and woke the event loop.

The callback on the listening socket fired: httpAccept().

▼ src/client_side.cc — Accepting connections
void
Comm::TcpAcceptor::acceptOne()
{
    // Pull one connection from the accept queue
    Comm::ConnectionPointer newConn = new Comm::Connection;
    newConn->fd = comm_accept(conn->fd, newConn->remote);

    if (newConn->fd < 0) {
        // EAGAIN: no more connections right now
        return;
    }

    // Set socket options: TCP_NODELAY, SO_KEEPALIVE, etc.
    commSetCloseOnExec(newConn->fd);
    fd_note(newConn->fd, "client http connect");

    // Pass the new connection to the protocol handler
    typedef CommCbFunPtrCallT<CommAcceptCbPtrFun> MyCall;
    AsyncCall::Pointer call = commCbCall(5, 5, "httpAccept",
                                          CommAcceptCbPtrFun(httpAccept));
    ScheduleCallHere(call);
}

static void
httpAccept(const CommAcceptCbParams &params)
{
    // Create ConnStateData to manage the new client connection
    ConnStateData *connState = new ConnStateData(params.conn, ...);
    connState->readNextRequest();
}

TcpAcceptor loops over the accept queue, pulling out all pending connections without blocking. For each one, it creates a ConnStateData — the Gatekeeper — and calls readNextRequest() to start reading the client's HTTP request.

A file descriptor was born. The kernel assigned it a number — say, 47. From this point forward, fd 47 represented one client's connection. It would live until the connection closed or timed out.

And a ConnStateData was born alongside it. The Gatekeeper. The object that would shepherd this connection from its first byte to its last.

Chapter 9 The First Visitor

The Gatekeeper called readNextRequest(). This registered a read callback on fd 47 with epoll. Then control returned to the event loop.

A few milliseconds later, epoll fired: the client had sent data. The Gatekeeper's read callback was scheduled. When it fired, bytes arrived in the read buffer.

GET http://www.example.com/index.html HTTP/1.1
Host: www.example.com
User-Agent: Mozilla/5.0 ...
Accept: text/html
Connection: keep-alive

Raw bytes in a buffer. Now they had to be parsed.

▼ src/client_side.cc — HTTP request parsing
void
ConnStateData::parseRequests()
{
    // Feed bytes into the HTTP/1 parser
    while (!inBuf.isEmpty()) {
        Http1::RequestParser hp;
        hp.parse(inBuf);

        if (hp.needsMoreData()) {
            // Incomplete request — wait for more bytes
            break;
        }

        if (hp.parseError()) {
            // Malformed request — send 400 Bad Request
            debugs(33, 2, "Bad HTTP request");
            clientStreamAbort(...);
            break;
        }

        // Complete request parsed
        // Build HttpRequest object from parsed fields
        HttpRequest *req = HttpRequest::FromUrlXXX(
            hp.requestUri(), hp.method(), hp.messageVersion());

        // Start processing the request
        ClientHttpRequest *http = new ClientHttpRequest(this);
        http->request = req;
        clientProcessRequest(this, hp, http);
    }
}

Http1::RequestParser is a state-machine parser. It handles incomplete data (TCP can split a request across multiple packets), malformed requests, and all the edge cases of HTTP/1.0 and HTTP/1.1. On success, it returns a parsed method, URI, version, and headers.

The parser worked through the bytes. GET — the method. http://www.example.com/index.html — the URI. HTTP/1.1 — the version. Then headers, one by one, until the blank line that marked the end of the header section.

A HttpRequest object materialized. The visitor's passport.

Chapter 10 Forging the Passport

The HttpRequest was not just a bag of strings. It was a richly structured object.

class HttpRequest : public Http::Message
{
public:
    HttpRequestMethod method;     // GET, POST, CONNECT, ...
    AnyP::Uri url;                // Parsed URL components

    // Scheme, host, port, path, query — all separated
    // url.getScheme() == AnyP::PROTO_HTTP
    // url.host()      == "www.example.com"
    // url.port()      == 80
    // url.path()      == "/index.html"

    HttpHeader header;            // Parsed headers
    Auth::UserRequest::Pointer auth_user_request; // Auth info

    // Cache-Control flags extracted from headers
    bool no_cache;
    bool no_store;

    // Connection flags
    bool persistent;              // Connection: keep-alive?
}

The URL parser was its own maze. AnyP::Uri::parse() handled the full complexity of RFC 3986 — scheme, authority, path, query, fragment. HTTPS CONNECT requests had a different form: CONNECT example.com:443 HTTP/1.1, where the "URL" was just a host:port pair. The parser had to know the difference.

Once the passport was forged, the visitor entered the kingdom's inner corridors. The callout chain.

Chapter 11 The World Within the Walls

The callout chain. doCallouts(). A series of checkpoints between "request received" and "response sent."

Each checkpoint was guarded. Some were always present. Some activated only under certain conditions. They ran in a fixed order, one at a time, each asynchronous — meaning each checkpoint could pause, wait for external information, and resume later.

▼ src/client_side_request.cc — The callout chain
void
ClientRequestContext::doCallouts()
{
    // Each flag tracks whether this callout has been done
    if (!http->request->flags.done_follow_x_forwarded_for) {
        checkFollowXForwardedFor();
        return;  // async — will call doCallouts() again when done
    }

    if (!http->request->flags.done_adaptation) {
        checkAdaptation();      // ICAP REQMOD check
        return;
    }

    if (!http->request->flags.done_acl_http_access) {
        http->request->flags.done_acl_http_access = true;
        // This is the main access control check
        ACLFilledChecklist *acl_checklist =
            clientAclChecklistCreate(Config.accessList.http, http);
        acl_checklist->nonBlockingCheck(
            clientAccessCheckDoneWrapper, this);
        return;
    }

    if (!http->request->flags.done_redirect) {
        checkRedirect();
        return;
    }

    // All callouts done — proceed to cache lookup
    clientProcessRequest2();
}

doCallouts() is a recursive-by-callback function. Each callout sets its flag when it completes and re-calls doCallouts(). The function effectively resumes from where it left off on each call. It's a hand-written coroutine — sequential logic expressed as a chain of callbacks.

The most important callout: ACL http_access check. This is where the Judge was summoned. The Gatekeeper handed the visitor's passport to the Judge and waited for a verdict: allowed or denied.

If denied, the connection received an error page and was closed. If allowed, the kingdom's next challenge awaited: the cache lookup.

Chapter 12 The Magic of Concurrency

Pause here. Step back.

Everything described so far — accepting a connection, reading bytes, parsing HTTP, running the callout chain — happened in a single thread, on a single CPU core. No parallelism.

And yet, while this visitor's ACL check was waiting for an external LDAP query to return, Squid was simultaneously:

  • Streaming a cached video to another client
  • Completing a TLS handshake with a third client
  • Receiving a DNS reply for a fourth client's request
  • Writing a completed cache entry to disk
  • Responding to a cache manager HTTP request

How? The event loop. Every operation that would block — waiting for a network reply, waiting for disk I/O, waiting for a timer — was converted into a non-blocking operation plus a callback. The callback was registered with epoll. The event loop drove all of it from a single thread.

I am not like a web server that spawns a thread per request. I am one. One thread, one process — in default mode — handling thousands of simultaneous connections. While I wait for your DNS reply, I'm also serving someone else's cache hit. While I'm writing your response to disk, I'm reading someone else's request off the wire. I never block. I only wait — and while I wait, I work on something else.

Chapter 13 The March of the Timers

Beyond request-driven events, the kingdom had its own internal clock.

Timers. Periodic events registered with eventAdd(). They fired whether or not any client was connected.

// Maintenance timers (from mainInitialize)
eventAdd("storeMaintain",      storeMaintain,      nullptr,  1.0, 1);  // every 1s
eventAdd("ipcache_purgelru",   ipcache_purgelru,   nullptr, 10.0, 1);  // every 10s
eventAdd("fqdncache_purgelru", fqdncache_purgelru, nullptr, 15.0, 1);  // every 15s
eventAdd("memPoolCleanIdlePools", MemPools::GetInstance().flushMeters,
                                               nullptr, 15.0, 1);  // every 15s
eventAdd("clientdbGC",         clientdbGC,         nullptr, 6*3600, 1); // every 6h

Each timer fired, did its work, and re-registered itself for the next interval. The system never stopped cleaning up after itself.

storeMaintain was the most frequent. Every second, it asked the cache: "Are you over capacity? Are any entries expired?" If yes, it evicted a batch. Small increments, continuously, so the cache never had to do a massive stop-the-world purge.

Chapter 14 SMP — The Kingdom Multiplied

Single-threaded was fast but had a ceiling. Modern servers had 16, 32, 64 cores. Squid's single thread could max out one core, but the other 63 sat idle.

The answer: SMP mode. workers N in squid.conf.

▼ squid.conf — SMP configuration
# Spawn 8 independent worker processes
workers 8

# Shared memory for cross-worker coordination
# (automatically managed by Squid)

# Each worker binds to the same port via SO_REUSEPORT
# The kernel load-balances incoming connections across workers
http_port 3128

In SMP mode, Squid spawns N worker processes plus one Coordinator process plus disk helper processes. Each worker runs a full event loop independently. They share a cache via shared memory (ipc/) and coordinate on in-progress requests via CollapsedForwarding.

The SMP architecture used separate processes, not threads. Each worker was an independent OS process with its own event loop, its own file descriptor table, its own stack. They shared nothing by default — except for the shared memory regions used for the transient cache and for coordination messages.

The Coordinator process managed the workers. It handled configuration reloads, reconfigure signals, and monitored worker health. If a worker crashed, the Coordinator could restart it without taking down the entire proxy.

CollapsedForwarding solved the thundering-herd problem across workers. If two workers both got a cache miss for the same URL simultaneously, only one would forward to the origin. The other would wait for the result in shared memory.

Chapter 15 At the Crossroads

The visitor's passport was forged. The callout chain had run. The ACL had spoken: allowed.

Now came the crossroads.

Two paths. Two questions:

First: Does the kingdom already have what this visitor wants? Check the cache. If yes — a cache hit — serve it immediately without leaving the building.

Second: If no — a cache miss — the Forward Commander must be dispatched beyond the border to retrieve it from the origin server.

The cache lookup was about to begin. The Storekeeper would be consulted. And the kingdom's memory — deep, vast, and finitely bounded — would answer.

I sit at the crossing between memory and the outside world. Every request either finds what it needs in my warehouse or makes me send someone to fetch it. My hit rate is my worth. Eighty-seven percent today. Eighty-seven packets out of a hundred never left this building. The origin servers never felt them. The network never saw them. They were mine to give. That is what I am for.

Volume One ends here.
