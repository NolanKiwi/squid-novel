VOLUME 2

The Court of the Judge

ACL, Rules, and Verdicts

The request passed through the gate. But even inside there are checkpoints. Judge ACL interrogates every packet. Allow or deny — the law breathes between zero and one.

Chapter 1

The Gate After the Gate

The Gatekeeper accepted the connection. Parsed the HTTP request. Read the method, URL, headers — all the way through. And then handed the work to a new actor: ClientHttpRequest.

ClientHttpRequest was not the Gatekeeper. It was not a guardian of raw connections. It was a processor of completed requests — a being that existed to carry one petition from the client's mouth to the kingdom's final verdict. And its first act was to march the petition to the courtroom.

The courtroom had a name: src/client_side_request.cc. Inside it a chain of tests waited — a sequence called doCallouts(). X-Forwarded-For processing. ICAP adaptation. And then, most importantly: the ACL judgment.

src/client_side_request.cc — doCallouts()
void
ClientHttpRequest::doCallouts()
{
    // Step 1: X-Forwarded-For
    if (!calloutContext->host_header_verify_done)
        hostHeaderVerify();
    // Step 2: ICAP
    else if (!calloutContext->adaptation_acl_check_done)
        adaptationAclCheckDone(...);
    // Step 3: ACL
    else if (!calloutContext->acl_checklist_done)
        aclCheckFastOrSlow();
    // ...
}

The chain was unbreakable. Each step had to complete before the next could begin. And if any step returned DENY — the petition died there, without ever reaching the cache or the network.

The courtroom awaited.

Chapter 2

The Judge's Constitution

Judge ACL's laws were not born at runtime. They were written in a file: squid.conf. Lines like:

acl localnet src 10.0.0.0/8
http_access allow localnet
http_access deny all

These declarations were the kingdom's constitution. When the kingdom booted, ConfigParser read the conf file from top to bottom and called upon a man named aclParseAclLine() to interpret each acl line.

aclParseAclLine looked at the type — src, dst, time, url_regex, and hundreds more — and called a factory function. The factory conjured the right ACL object and stored it in a global map keyed by name.

src/acl/Acl.cc — ACL factory
Acl::Node *
Acl::Node::Make(const char * const typeName)
{
    const auto & makers = Registry();
    const auto pos = makers.find(typeName);
    if (pos == makers.end())
        throw TextException(ToSBuf("unknown ACL type: ", typeName), ...);
    return (pos->second)();
}

Each ACL type registered itself at program startup using a static initializer. The registry was a map from type name to factory lambda. When the constitution was read, the factories breathed life into an army of judgment objects — each one a frozen rule waiting to be consulted.

A law that no one reads is still a law. These objects slept in memory for the entire lifetime of the proxy, awakened only when a request knocked on the courtroom door.
Chapter 3

The Checklist

When a request arrived at the courtroom, the system did not invoke ACL objects directly. It created a temporary document — a ACLChecklist — and handed it the request's context.

The Checklist was the trial itself. It held a reference to the request, carried the client's IP address, remembered which rules had been consulted and which had not, and maintained a stack for the current evaluation path through nested AND/OR logic.

src/acl/Checklist.h — the trial document
class ACLChecklist {
public:
    Acl::Answer fastCheck();
    void nonBlockingCheck(ACLCB *, void *);

private:
    const acl_access *accessList = nullptr;
    Acl::InnerNode::Breadcrumbs matchPath;
    bool asyncInProgress_ = false;
    // ...
};

The Checklist had two modes of operation. The first was fastCheck() — synchronous evaluation, completed within the current call stack. The second was nonBlockingCheck() — asynchronous evaluation, which could suspend mid-trial while waiting for an external process to respond, and resume later via callback.

The distinction mattered enormously. Most ACL types were fast: IP matching, time checking, string matching. But some ACL types had to consult external oracles — processes outside the proxy that knew things the proxy did not. Those required the async path, and the trial would be suspended until the oracle answered.

A trial that cannot pause is a trial that cannot be fair. The Checklist learned to wait.
Chapter 4

The IP Ledger

The most common ACL type in the kingdom was src — match by source IP address. Its implementation lived in src/acl/Ip.cc, and it used a data structure older than most of its colleagues: the radix trie.

A radix trie is a tree where each node represents a bit of the address. To look up whether 192.168.1.42 was in the set 192.168.0.0/16, the trie walked 16 bits down the tree and found a match. To check whether it was NOT in any of ten thousand blocked ranges, it still walked only 32 bits — the depth of an IPv4 address.

src/acl/Ip.cc — radix lookup
int
ACLIP::match(ACLChecklist *checklist)
{
    const auto & ip = dynamic_cast<ACLIPChecklist*>(checklist)->src_addr;
    return data->match(ip);
    // data is a SplayTree<acl_ipdata> with radix semantics
}

IPv6 was handled identically, with 128 bits instead of 32. The same radix structure, the same walking logic, the same O(address-length) lookup time regardless of how many addresses were in the set.

The IP Ledger was the fastest judge in the courtroom. No disk access, no network calls, no external processes. Just a tree walk through bits already in memory. For the vast majority of requests, it rendered its verdict in microseconds.

The best law is the law that answers before you finish asking the question.
Chapter 5

The Clock Court

Some laws applied only at certain hours. The time ACL type, implemented in src/acl/Time.cc, was the kingdom's temporal gatekeeper — a judge who looked not at who was asking, but when they were asking.

The implementation was elegant in its simplicity. For each configured time range, the ACL stored a bitmask — one bit for each minute of the week. There are 7 × 24 × 60 = 10,080 minutes in a week. The bitmask was exactly 10,080 bits — 1,260 bytes.

src/acl/Time.cc — weekly bitmask
// "time MTWHF 08:00-17:00" compiles to:
// bit[Monday*1440 + 8*60 + 0] through [Monday*1440 + 16*60 + 59] = 1
// bit[Tuesday*...] through [...] = 1
// ... and so on for W, T, H, F

int
ACLTime::match(ACLChecklist *checklist)
{
    const time_t when = squid_curtime % (7 * 24 * 3600);
    const int minute = when / 60;
    return data->find(minute) ? 1 : 0;
}

To check whether the current time matched a time ACL, the code computed which minute of the week it currently was — a number from 0 to 10,079 — and looked up that bit. One bit test. No parsing, no date math, no string comparison. The entire schedule was precompiled at config time into a lookup table that answered in nanoseconds.

The Clock Court never deliberated. It glanced at its watch and immediately knew.

Chapter 6

The Pattern Witch

Not all judgments could be expressed as IP ranges or time windows. Some required pattern matching — the ability to say "this URL looks like it contains the word 'malware'" or "this domain ends in .evil.com". For this the kingdom employed the Pattern Witch: ACLRegex.

The Pattern Witch lived in src/acl/RegexData.cc and carried a compiled list of regular expressions. Each expression was a POSIX regex, compiled at config time and stored ready-to-execute.

src/acl/RegexData.cc — regex matching
bool
ACLRegexData::match(const char *word)
{
    for (const auto & re : data) {
        if (re.match(word)) {
            debugs(28, 4, "regex match: " << re.c_str());
            return true;
        }
    }
    return false;
}

The Pattern Witch was powerful but expensive. A list of a thousand regexes tested against every URL meant a thousand regex evaluations per request. For this reason, experienced administrators kept their regex ACLs sparse — or replaced them with faster alternatives like dstdomain, which used a hash lookup instead of pattern matching.

But for the cases where only a pattern would do — URL blacklists, content type filters, custom header inspections — the Pattern Witch was indispensable. She saw meaning in strings where the other judges saw only bytes.

She whispered the patterns to herself as she read, and the patterns whispered back: match, no match, match.
Chapter 7

The Outer Judges

There was knowledge the kingdom could not hold within itself. User databases, active directory servers, custom business logic systems — these lived outside the proxy and could not be compiled into squid.conf. To consult them, the kingdom employed a special mechanism: external ACL.

The external ACL system, implemented in src/external_acl.cc, spawned helper processes — small programs that read lines from stdin and wrote OK or ERR to stdout. The proxy sent them formatted queries and waited for answers.

external ACL helper protocol
# squid sends:
192.168.1.42 jsmith GET http://example.com/

# helper responds:
OK user=jsmith group=employees log="authenticated"
# or:
ERR message="user not found"

But waiting was expensive. The proxy cached external ACL responses — keyed by the query string, with a configurable TTL. If the same user from the same IP requested the same resource within the TTL window, the cached answer was used and the helper was not consulted.

The cache was a bounded hash table with LRU eviction. It stored ExternalACLEntry objects, each with an answer, a timestamp, and a tag — whether the answer was still fresh, nearly stale, or expired.

The outer judges were wise but slow. The kingdom learned to remember their verdicts.
Chapter 8

The Path Recognizer

URL structure was another axis of judgment. The url_path, urlpath_regex, and dstdomain ACL types allowed administrators to make decisions based on the destination of a request, not its source.

The dstdomain type was special. It used a hash set rather than a list or tree, enabling O(1) average-case lookup. When compiled, the list of domains was stored in a SplayTree sorted lexicographically from right to left — so that .example.com entries would match both www.example.com and mail.example.com.

src/acl/DomainData.cc — right-to-left domain matching
bool
ACLDomainData::match(const char *host)
{
    // compare reversed: "moc.elpmaxe" matches ".moc.elpmaxe"
    char reversed[SQUIDHOSTNAMELEN];
    reverseString(host, reversed);
    return data.find(reversed) != nullptr;
}

This design meant that a wildcard entry like .example.com stored as moc.elpmaxe. would match any hostname whose reversed form started with moc.elpmaxe. — which was exactly any subdomain of example.com. The dot prefix became a prefix match in the reversed space.

The Path Recognizer knew destinations. The IP Ledger knew origins. Together they defined the two-dimensional space of access control: who is asking, and where are they trying to go.

Chapter 9

The Identity Inquisitors

The kingdom demanded identity from its subjects. The proxy_auth ACL type required HTTP authentication — Basic, NTLM, Kerberos/Negotiate — before any request could proceed. The authentication subsystem lived in src/auth/.

When a request arrived without credentials, the ACL evaluation did not simply return DENY. It returned a special status — AUTH_REQUIRED — which caused the Gatekeeper to send a 407 Proxy Authentication Required response back to the client, advertising which schemes were available.

src/auth/basic/Config.cc — Basic auth challenge
Auth::UserRequest::Pointer
Auth::Basic::Config::decode(char const *proxy_auth, ...)
{
    // parse "Basic dXNlcjpwYXNz" (base64)
    auto *result = new Auth::Basic::UserRequest();
    // decode credentials, look up in helper cache
    return result;
}

NTLM and Kerberos were stateful protocols requiring multiple round-trips. The proxy stored authentication state per TCP connection — because the authentication handshake was bound to the connection, not the request. A connection that completed NTLM negotiation was marked as authenticated; subsequent requests on that connection skipped the challenge.

The Identity Inquisitors were relentless. No one passed without a name.

A credential is a key. But a key can be copied. The inquisitors verified not just that you had the key, but that it was yours.
Chapter 10

The Logic Architects

Simple ACL types — IP, time, domain, regex — were the atoms of the judgment system. But the kingdom also needed to compose them: allow access if the source is local AND the time is business hours AND the URL is not in the blacklist. This compositional logic was the work of the Logic Architects.

The architects lived in src/acl/AllOf.cc, AnyOf.cc, and Not.cc. They were the boolean operators of the ACL system: AND, OR, NOT. Each was itself an ACL node that contained other ACL nodes as children.

src/acl/AllOf.cc — AND logic with short-circuit
Acl::Answer
Acl::AllOf::matches(ACLChecklist *checklist) const
{
    for (const auto & node : nodes) {
        const auto result = node->matches(checklist);
        if (!result.allowed())
            return result; // short-circuit: first false ends it
    }
    return Acl::Answer(ACCESS_ALLOWED);
}

The evaluation was short-circuit. An AND chain stopped at the first DENY. An OR chain stopped at the first ALLOW. This meant that expensive ACL types — like external ACL — could be placed at the end of a chain, and would only be consulted if the cheaper tests hadn't already resolved the question.

But short-circuit had a subtlety: asynchronous ACL types could not simply "stop" — they had to suspend the entire checklist. The Logic Architects maintained a matchPath breadcrumb trail so that when an async ACL resumed, the evaluation could continue from exactly where it had paused.

Chapter 11

Asynchrony — The Suspended Trial

The most complex aspect of the ACL system was not any individual judge. It was the mechanism of suspension — how a trial could pause mid-evaluation, yield control back to the event loop, and resume later without losing its place.

When an asynchronous ACL was encountered — say, an external ACL helper was being queried — the Checklist set a flag: asyncInProgress_ = true. The evaluation function returned without a final answer. Control returned to the event loop.

src/acl/Checklist.cc — async suspension and resumption
void
ACLChecklist::checkCallback(Acl::Answer answer)
{
    asyncInProgress_ = false;
    if (finished()) {
        // trial is over, fire the callback
        checkNB_callback(this, answer);
        return;
    }
    // resume the trial from where it paused
    matchAndFinish();
}

The suspended trial was vulnerable. While it waited, the world could change. The connection could close. The request could be aborted. The Checklist used CbcPointer to detect this — the same use-after-free prevention mechanism that protected all async jobs in the kingdom. If the Checklist's owner had vanished while it waited, the resumption detected this and quietly dissolved the trial rather than operating on dead memory.

The suspended trial was a ghost. It might return to a courtroom that no longer existed. The law had to account for this.
Chapter 12

The Penalty Scribes

When the Checklist reached its verdict — DENY — the trial was not simply over. The denial had to be communicated to the client. This was the work of the Penalty Scribes: the error page generation system in src/errorpage.cc.

An ErrorState object was created, carrying the error type — ERR_ACCESS_DENIED, ERR_CACHE_ACCESS_DENIED, ERR_TOO_BIG, and dozens more. The error type mapped to an error page template, usually an HTML file stored in the system's error directory.

src/errorpage.cc — error template expansion
// Template variables like %U (URL), %e (errno), %c (Squid error code)
// are expanded at response-generation time
MemBuf
ErrorState::buildBody()
{
    const char *m = template_.c_str();
    // scan for % escape sequences, expand each one
    // %U → request URL, %w → wait time, %z → DNS error detail
}

The error pages were customizable per language, per error type, and even per ACL. Administrators could deploy completely custom HTML with their organization's logo, help desk number, and a message explaining why the site was blocked.

The Penalty Scribes took no pleasure in their work. But they were thorough. Every denial was documented, every error code precise, every response properly formatted with the correct HTTP status code.

Chapter 13

The Glass Inspector

Encrypted traffic was an ancient problem of the courtroom. How could Judge ACL render a verdict on a request it could not read? HTTPS connections arrived as opaque tunnels — the Gatekeeper could see the TCP connection but not the HTTP inside it.

The solution was SSL Bump, implemented in src/ssl/ and src/security/. SSL Bump was a man-in-the-middle technique: the proxy presented its own certificate to the client, terminated the TLS session, read the plaintext, made its ACL decision, then re-encrypted and forwarded to the server.

SSL Bump three-step protocol
// Step 1: peek — read ClientHello, extract SNI hostname
// ssl_bump peek step1
// → now we know the hostname, can apply dstdomain ACLs

// Step 2: peek — complete TLS handshake with server
// ssl_bump peek step2
// → now we can see server cert, apply cert-based ACLs

// Step 3: splice or bump
// ssl_bump splice — pass through without decryption
// ssl_bump bump  — decrypt, inspect, re-encrypt

The Glass Inspector could see through HTTPS. But this power came with responsibility — and controversy. SSL Bump required installing the proxy's CA certificate on every client device so they would trust the proxy's forged certificates. In corporate environments this was standard practice. In public networks it was deeply problematic.

To inspect the letter, you must open it. The Glass Inspector was always aware of this cost.
Chapter 14

A Day in the Courtroom

Let us follow a single request through the full courtroom sequence.

A client at 10.0.1.42 requested http://news.example.com/article/12345 at 14:32 on a Wednesday. The request arrived at the Gatekeeper, was parsed, and passed to ClientHttpRequest, which called doCallouts().

First checkpoint: X-Forwarded-For. The request had no XFF header, so no action was needed.

Second checkpoint: ICAP. The configuration had no ICAP services, so this was skipped.

Third checkpoint: ACL evaluation. The access list was:

acl localnet src 10.0.0.0/8
acl business_hours time MTWHF 08:00-18:00
acl blocked_sites dstdomain .gambling.com .adult.com
http_access deny blocked_sites
http_access allow localnet business_hours
http_access deny all

The Checklist was created. It walked the access list from top to bottom.

Rule 1: deny blocked_sites. Is news.example.com in the blocked_sites list? The Path Recognizer consulted its domain hash. No match. This DENY rule did not apply.

Rule 2: allow localnet business_hours. Is 10.0.1.42 in 10.0.0.0/8? The IP Ledger walked the radix trie. Yes. Is Wednesday 14:32 in business hours? The Clock Court checked the bitmask. Yes. Both conditions met — ALLOW.

Verdict: ALLOW. The trial was 200 microseconds. The request proceeded to the cache.

Justice at this speed is not justice. It is architecture.
Chapter 15

The Law That Never Ends

The courtroom was not a place you visited once and left. Every request passed through it. Millions of requests per day, each one submitted to the same sequence of judges, each one receiving a verdict within microseconds.

The ACL system had been accumulating laws since Squid's earliest days. Some ACL types were thirty years old. Others — like ssl::server_name for SNI-based SSL Bump decisions — had been added in the past decade. New ACL types were still being added. The law grew.

What kept the courtroom from collapsing under its own weight was discipline in design. Each ACL type was a small, focused object with a single responsibility. The composition logic was separate from the individual judges. The async machinery was separate from the evaluation logic. Layers of abstraction that had held for decades.

Acl::Node hierarchy — the type tree
Acl::Node
├── Acl::InnerNode  (AllOf / AnyOf / NotNode — logic composition)
│   ├── Acl::AllOf
│   ├── Acl::AnyOf
│   └── Acl::NotNode
└── Acl::LeafNode   (actual data matchers)
    ├── ACLIP       (src / dst)
    ├── ACLTime     (time)
    ├── ACLRegex    (url_regex / etc.)
    ├── ACLDomain   (dstdomain)
    └── ACLExternal (external_acl_type)

The kingdom's law was a tree. Not a flat list, not a monolithic function, but a recursive structure where every node could contain other nodes. A request was evaluated by walking this tree, and the tree could grow arbitrarily complex without changing the walking algorithm.

Judge ACL would outlast the current administrators, the current hardware, the current generation of threats. The law was designed to outlast its authors.

Laws are not written for today. They are written for every day that follows — including the days no one has yet imagined.
