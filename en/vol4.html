Beyond the Border

Forwarding, Connections, and Tunnels

Cache miss. The Forward Commander steps up. Consults the DNS Oracle, shouts to the peer caches, races IPv6 against IPv4, and sometimes digs a secret tunnel.

Chapter 1

The Cache Miss

The Storekeeper had checked all three vaults. Memory: nothing. Disk: nothing. The object had never been fetched, or it had been fetched and forgotten. Either way, the answer was the same: MISS.

The Freshness Oracle confirmed it — there was nothing to validate, nothing to refresh. The request had to leave the kingdom's borders and go to the open internet to fetch what was needed.

This moment — the cache miss — was the proxy's most expensive operation. It required DNS resolution, TCP connection establishment, TLS negotiation, an HTTP request, a response, and the streaming of potentially megabytes of data back through the proxy to the client. Every cache hit avoided all of this. Every miss forced it.

src/client_side_request.cc — beginning the forward path
void
ClientHttpRequest::processMiss()
{
    // Create a new StoreEntry for this miss
    http->storeEntry(storeCreateEntry(...));

    // Hand off to the Forward Commander
    FwdState::Start(
        http->getConn()->clientConnection,
        http->storeEntry(),
        http->request,
        http->al
    );
}

The Forward Commander received the baton. The mission had begun.

Chapter 2

The Forward Commander

FwdState, implemented in src/FwdState.cc, was the Forward Commander — the coordinator of everything required to fetch a response from beyond the proxy's borders. Its job was to orchestrate a complex sequence of async operations and deliver a result to the waiting StoreEntry.

The Commander's lifecycle had a clear progression: peer selection → DNS resolution → connection establishment → HTTP request dispatch → response streaming → completion (or failure and retry).

src/FwdState.cc — state machine overview
class FwdState {
    // Phase 1: Where do we send this?
    void peerSelect();           // ask neighbors.cc

    // Phase 2: What is the IP address?
    // (done implicitly by HappyConnOpener via ipcache)

    // Phase 3: Connect
    void connectStart();         // spawn HappyConnOpener
    void noteConnection(...);    // connection established callback

    // Phase 4: Send request, receive response
    void dispatch();             // hand to serverConnection

    // Phase 5: Done or retry
    void complete();
    void reforward();
};

The Commander was an AsyncJob — it registered itself with the event loop, yielded control between phases, and was garbage-collected by the RefCount machinery when its mission ended. It held a CbcPointer to the StoreEntry to detect if the entry had been purged while the forward was in progress.

The Forward Commander was always calm. Whether the mission took 50 milliseconds or 50 seconds, it followed the same sequence: one step at a time, notify when done, handle failure without panic.
Chapter 3

The Peer Council

Before going directly to the origin server, the Forward Commander consulted the Peer Council: the neighbor cache hierarchy defined in src/neighbors.cc.

Squid supported two types of peers: parents (caches higher in the hierarchy that could both cache and forward) and siblings (caches at the same level that could only share cached content). The peer selection algorithm consulted each configured peer and asked: do you have this object?

src/neighbors.cc — peer query protocols
// ICP (Internet Cache Protocol) — UDP, fast, unreliable
// Send ICP_QUERY to all peers, wait for ICP_HIT or ICP_MISS
// Use first HIT peer, or parent if all MISS

// HTCP (Hyper Text Caching Protocol) — UDP, more features
// Like ICP but with full HTTP headers in query/response
// Supports CLR (clear) operation for purge propagation

void
neighborsUDPPing(HttpRequest *request, StoreEntry *entry,
                 IRCB *callback, void *data, ...)

Peer selection had to be fast — clients were waiting. The UDP-based ICP protocol was designed for this: send a small query to multiple peers simultaneously, wait a configurable number of milliseconds for responses, pick the best peer that responded. If no peer had the object, go directly to the origin.

The Peer Council represented the internet-age vision of cooperative caching: a mesh of proxies that collectively reduced load on origin servers by sharing their caches. In modern deployments this had largely been replaced by CDN architectures, but the machinery remained in Squid for organizations that still used hierarchical proxy configurations.

Chapter 4

The DNS Oracle

A hostname was not an address. Before the Forward Commander could open a TCP connection to www.example.com, it needed to know the IP address — and for that it needed to consult the DNS Oracle: src/ipcache.cc.

The DNS Oracle was an asynchronous DNS resolver with a built-in cache. It maintained a hash table of hostname → [IP addresses] mappings, each with a TTL. If a hostname had been recently resolved, the cached answer was returned immediately. If not, a real DNS query was dispatched.

src/ipcache.cc — async DNS with multi-IP support
void
ipcache_nbgethostbyname(const char *name, IPH *handler, void *handlerData)
{
    IpcacheEntry *i = ipcache_get(name);
    if (i && !ipcacheExpired(i)) {
        // cache hit — call handler immediately (async convention)
        ipcacheCallback(i, Ip::Forward, -1, handler, handlerData);
        return;
    }
    // cache miss — dispatch DNS query, callback when resolved
    ipcacheAddPending(name, handler, handlerData);
}

The DNS Oracle returned not just one IP address but all of them — the full list returned by the DNS server. This was critical for Happy Eyeballs (the next chapter) and for failover. If a connection to the first IP failed, the Forward Commander could try the second, the third.

Negative results were also cached — NXDOMAIN responses (hostname does not exist) and SERVFAIL results were stored with short TTLs to prevent storms of DNS queries for known-bad hostnames.

The DNS Oracle knew every name the kingdom had ever asked about. It was the oldest and most consulted advisor in the court.
Chapter 5

The Address Book

DNS was not the only source of addresses. The kingdom maintained additional address resolution mechanisms for special cases.

ipcache was the primary cache, storing DNS results. But src/fqdncache.cc maintained the reverse: a cache of IP → hostname mappings for reverse DNS lookups, used in access logging and some ACL types.

The hosts file was also consulted — Squid could be configured to read the system's /etc/hosts file and use those static mappings before querying DNS. Useful for test environments, split-horizon DNS, and overriding DNS for specific hostnames.

src/dns/rfc1035.cc — DNS packet handling
// Squid includes its own minimal DNS client
// (not relying on getaddrinfo, which is blocking)
// Sends UDP queries to configured nameservers
// Handles: A, AAAA, PTR, CNAME (chain following)
// Timeout and retry logic for unreliable UDP

ssize_t
rfc1035BuildAQuery(const char *hostname, char *buf, size_t sz,
                   unsigned short qid, rfc1035_query * query)

Squid's DNS client was deliberately minimal — it handled only the record types needed for proxy operation. Full DNS functionality (MX records, SRV records, DNSSEC) was not implemented. This kept the code small and focused, at the cost of flexibility for advanced configurations that needed non-standard DNS behavior.

Chapter 6

Happy Eyeballs

The DNS Oracle had returned multiple addresses — perhaps two IPv6 and three IPv4. Which one should the Forward Commander connect to? The naive answer was "try them in order." The correct answer was far more sophisticated.

RFC 8305 defined Happy Eyeballs, an algorithm designed to find the fastest working connection without making the user wait for sequential failures. The Squid implementation lived in src/HappyConnOpener.cc.

src/HappyConnOpener.cc — RFC 8305 racing
// Step 1: Try IPv6 first (RFC 8305 §4)
connectToNextPeer(); // attempt IPv6

// Step 2: If IPv6 doesn't connect within 250ms...
if (waitingForSpeedRace && gotNewAddress) {
    // Start IPv4 attempt in parallel — don't cancel IPv6
    connectToNextPeer(); // attempt IPv4
}

// Step 3: Use whichever connects first
// Cancel the slower one

The 250ms timeout before launching the parallel IPv4 attempt was calibrated to prefer IPv6 (which is generally preferred by RFC 8305) while not penalizing users with broken IPv6 connectivity by making them wait for a full IPv6 timeout. If IPv6 connected within 250ms, IPv4 was never tried. If it didn't, both raced.

The name "Happy Eyeballs" came from the user experience goal: users should not see a "spinning" browser indicator caused by network protocol fallback. Their experience should remain smooth — happy — regardless of which protocol worked.

Competition between protocols was not waste. It was the algorithm's way of serving the user better than either protocol could alone.
Chapter 7

The Connection Pool

TCP connection establishment was expensive: SYN, SYN-ACK, ACK — a round trip before a single byte of HTTP could be sent. For TLS connections, add two more round trips for the handshake. For short-lived HTTP requests on busy servers, this overhead dominated total latency.

The solution was connection reuse. Squid maintained a connection pool — a cache of idle established connections, organized by destination (hostname, port, TLS parameters). When the Forward Commander needed a connection to a server it had recently communicated with, it checked the pool first.

src/servers/FwdServer.cc — connection pool
Comm::ConnectionPointer
FwdState::pconnPop(const Comm::ConnectionPointer &dest, const char *domain)
{
    // Check idle connection pool
    Comm::ConnectionPointer temp =
        theOutgoingTransport.pop(dest, domain, checkReturnedConnectivity);
    if (temp != nullptr) {
        ++n_tries;
        return temp; // reuse existing connection
    }
    return nullptr; // must establish new connection
}

The pool had limits — a maximum number of idle connections per destination, and a global maximum. Connections too old or from destinations that hadn't been visited recently were closed and removed. The pool was managed by the same event loop that managed everything else — a periodic callback walked the pool and closed expired connections.

HTTP/1.1's Connection: keep-alive was the enabling protocol feature. HTTP/2 went further — a single TCP connection could carry hundreds of simultaneous request/response streams, making connection pooling even more valuable.

Chapter 8

The HTTP Negotiator

With a connection established, the Forward Commander handed control to a new actor: the HTTP Negotiator. This was the component responsible for sending the actual HTTP request and reading the response.

For HTTP/1.1, this was HttpStateData in src/http.cc — a state machine that wrote the request headers, optionally wrote a request body, read the response status line, read response headers, and then streamed the response body into the StoreEntry.

src/http.cc — request serialization
void
HttpStateData::sendRequest()
{
    MemBuf mb;
    mb.init();
    // Write request line and headers
    request->pack(&mb);
    // Send via the established connection
    Comm::Write(serverConn, &mb, callbackContext);
}

void
HttpStateData::processReply()
{
    // Parse response status and headers
    // Start streaming body to StoreEntry
    // Notify waiting clients (StoreClients) as data arrives
}

The HTTP Negotiator had to handle a large number of edge cases: chunked transfer encoding, gzip content encoding (optionally decoded by the proxy), HTTP/1.0 vs HTTP/1.1 differences, response headers that affected cacheability, redirects that required starting over with a new request.

For HTTP/2 connections, a different implementation — Http2::Stream — handled the multiplexed stream protocol, translating between Squid's internal request/response model and the HTTP/2 framing layer.

Chapter 9

The Secret Tunnel

Not all connections could be proxied as HTTP requests. HTTPS destinations — port 443 — used TLS from the very first byte. The proxy could not read the request URL, could not make ACL decisions on the request path, could not cache the response. It could only relay bytes.

This relay was implemented in src/tunnel.cc by the TunnelStateData class. When a client sent a CONNECT request:

CONNECT www.example.com:443 HTTP/1.1

— the proxy established a TCP connection to www.example.com:443 and then responded to the client:

HTTP/1.1 200 Connection established

After that, all data flowing between client and server was blindly relayed. The proxy copied bytes from client to server and from server to client without reading or modifying them.

src/tunnel.cc — TunnelStateData blind relay
// Two AsyncJobs connected back-to-back:
// client socket → server socket
// server socket → client socket

static void
copyRead(TunnelStateData *tunnelState, TunnelStateData::Connection &from)
{
    // Read from source
    comm_read(from.conn, from.buf, from.bytesWanted(...), copyReadHandler, ...);
}

static void
copyWrite(TunnelStateData *tunnelState, TunnelStateData::Connection &to)
{
    // Write to destination
    Comm::Write(to.conn, to.buf, to.len, copyWriteHandler, ...);
}

The tunnel was opaque. The proxy could not see what URLs were being fetched, could not cache responses, could not modify requests. ACL decisions on CONNECT requests had to be made on hostname and port alone — before the tunnel was established.

The Secret Tunnel was the proxy's admission that it could not always see. Sometimes the best service was to step aside and let the encrypted conversation flow through unexamined.
Chapter 10

The Failure Path

The Forward Commander's mission did not always succeed. DNS resolution might fail. The TCP connection might be refused. The TLS handshake might fail due to a certificate error. The server might respond with a 5xx status. The connection might drop mid-response.

Every failure was handled by the same error propagation machinery. The failing component called ErrorState::sendError() or invoked the Forward Commander's fail() method. The Commander recorded the error and decided: retry or give up?

src/FwdState.cc — failure handling
void
FwdState::fail(ErrorState *errorState)
{
    err = errorState;
    if (canRetry(errorState)) {
        reforward();    // try again with next peer/address
    } else {
        complete();     // mission failed — send error to client
    }
}

bool
FwdState::canRetry(ErrorState *err) const
{
    // Retryable if: not yet tried all peers/addresses
    //               AND error is retriable (connect refused,
    //               not read errors mid-response)
    return n_tries < max_tries && err->type != ERR_READ_ERROR;
}

When all retries were exhausted, the Forward Commander called complete() with the most informative error state it had collected. The error was passed to the ErrorState machinery, which generated an appropriate HTTP error response and delivered it to the client.

The client received a well-formatted error page — "Connection refused," "DNS resolution failed," "Gateway timeout" — rather than a raw protocol error. The kingdom maintained its composure even in failure.

Chapter 11

The Retry Logic

Retrying was not the same as repeating. A retry had to make a different choice — a different peer, a different IP address, a different transport — or it was not a retry but merely the same failure twice.

The Forward Commander tracked which peers and IP addresses had been tried and failed. Each retry took the next untried option from the peer selection and IP address lists. If a connection attempt to IPv6 address 1 of a server failed, the retry might try IPv6 address 2, or fall back to IPv4, or try a parent cache instead.

src/FwdState.cc — structured retry progression
void
FwdState::reforward()
{
    // Mark current server connection as failed
    if (serverConn) {
        // Don't return failed connection to pool
        serverConn->close();
        serverConn = nullptr;
    }

    // Request a new peer/address selection
    // (peerSelect will skip already-tried options)
    peerSelect();
}

For idempotent requests — GET, HEAD, OPTIONS — retrying after certain failures was safe. A GET request that failed mid-response could be retried from the beginning; the client would not know. For non-idempotent requests — POST, PUT, DELETE — retrying was dangerous. Retrying a POST that had partially succeeded could cause duplicate actions.

The retry logic was conservative for non-idempotent requests: retry only on connection failures before any request data was sent. Once the request had been transmitted, no retry.

Chapter 12

TLS Beyond the Gate

When the destination server required TLS — and by 2020 most did — the Forward Commander had to negotiate a TLS session. This was handled by the Security subsystem in src/security/.

TLS negotiation for outbound connections was distinct from TLS with clients. For outbound connections, Squid was the TLS client — it presented the server's certificate for verification, negotiated a cipher suite, and established the encrypted channel.

src/security/PeerConnector.cc — TLS handshake
class Security::PeerConnector : public AsyncJob
{
    // Phase 1: connect TCP (done by HappyConnOpener)
    // Phase 2: TLS handshake
    void negotiate();

    void handleNegotiateError(int result) {
        // SSL_connect returned <= 0
        // Could be: SSL_ERROR_WANT_READ (not an error, need more data)
        //           SSL_ERROR_SSL (fatal TLS error)
        //           SSL_ERROR_SYSCALL (TCP error during TLS)
    }
};

Certificate verification was a complex topic. By default, Squid verified the server's certificate against the system's CA bundle. But administrators could configure custom CA certificates, disable verification entirely (for internal servers with self-signed certs), or implement custom verification callbacks.

For SSL Bump deployments where the proxy was also decrypting client traffic, the outbound TLS session had to be established and inspected before re-encrypting for the client — a complex dance of two simultaneous TLS sessions for each HTTPS connection.

Chapter 13

Pinned Connections

Some protocols broke the proxy's clean separation between client connection and server connection. NTLM authentication was the most common: the NTLM handshake was bound to a specific TCP connection. If the proxy routed NTLM requests through different server connections, the handshake would fail.

The solution was connection pinning. Once an NTLM handshake began on a specific client connection → server connection pair, those two connections were pinned together. All subsequent requests from that client, for the duration of that authentication session, were sent on the same server connection.

src/client_side.cc — connection pinning
// ConnStateData tracks the pinned server connection:
Comm::ConnectionPointer pinning.serverConnection;
const char *pinning.host;
int pinning.port;
bool pinning.auth; // pinned because of auth handshake

// When serving a new request on this client connection:
// Check if we have a pinned server connection
// If yes, use it directly — skip peer selection, DNS, pool

Pinning was a pragmatic compromise. It sacrificed the proxy's ability to load-balance across server connections for the sake of protocol compatibility. Without it, NTLM authentication simply did not work through the proxy. With it, it worked, at the cost of slightly reduced connection efficiency for NTLM users.

The proxy could not change the protocols it served. It could only adapt to them, however inelegant the adaptation.
Chapter 14

The Mission Complete

The response arrived. Status 200. Content-Type: text/html. Content-Length: 48,293. The HTTP Negotiator read the headers, assessed cacheability, and began streaming the body into the StoreEntry.

As bytes arrived from the server, they were simultaneously written to the StoreEntry's memory buffer and read by the StoreClients — the waiting client connections. The client saw data flowing almost immediately, long before the full response had been received from the server.

When the last byte arrived and the response was complete, the StoreEntry's status changed from STORE_PENDING to STORE_OK. The disk write — if the object was cacheable and within size limits — had been running in parallel. The Storekeeper marked the entry as fully written.

src/FwdState.cc — mission complete
void
FwdState::complete()
{
    debugs(17, 3, "FwdState::complete " << entry->url());

    if (entry->store_status == STORE_PENDING) {
        entry->complete(); // mark as fully received
    }

    if (Comm::IsConnOpen(serverConn)) {
        if (entry->keepalive())
            pconnPush(serverConn, ...); // return to pool
        else
            serverConn->close();
    }
    // FwdState's RefCount drops to zero → self-destructs
}

The Forward Commander's final act was to either return the server connection to the pool (if keep-alive) or close it. Then the Commander's reference count dropped to zero, and it dissolved — its job done, its memory freed, its mission recorded in the StoreEntry it left behind.

Chapter 15

The Road That Always Opens

The forward path was Squid's most complex subsystem — a web of interacting async jobs, each responsible for one piece of the journey from "miss" to "response." DNS resolution, peer selection, Happy Eyeballs racing, connection pooling, HTTP negotiation, error handling, retry logic, TLS, pinning — each was a separate component, each communicating through callbacks and shared state.

What held this complexity together was the invariant of the forward path: either a response is delivered to the StoreEntry, or an error is delivered, but one or the other always happens. The Forward Commander guaranteed this. No silent failures, no hung requests, no resources leaked in the failure path.

Forward path component interaction
ClientHttpRequest::processMiss()
  └── FwdState::Start()
        ├── peerSelect() → neighbors.cc
        ├── HappyConnOpener
        │     ├── ipcache (DNS)
        │     └── Comm::Connect (TCP)
        ├── Security::PeerConnector (TLS)
        ├── HttpStateData (HTTP/1.1)
        │     └── StoreEntry stream
        └── complete() / fail()

The road to the open internet was never blocked forever. DNS servers went down, network routes failed, origin servers crashed — but Squid always found a path or honestly reported that no path existed. The mission might fail, but it would not be forgotten. The kingdom's honor required a clear answer, in both success and defeat.

The Forward Commander did not always return victorious. But it always returned. There was no going out and simply not coming back. Every mission had an ending — and an accounting.
