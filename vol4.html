Volume 4

국경 너머로

포워딩, 연결, 그리고 터널

창고에 없다. 캐시 미스.
선발대장이 나선다. DNS에게 묻고, 피어에게 외치고,
행복하게 연결하고, 때로는 땅굴을 판다. 인터넷 너머로.

제 1 장 선발대장의 소명

캐시 미스. 창고지기가 고개를 저었다. "없어요. 직접 가져와야 해요."

이 순간을 기다린 자가 있었다. 선발대장 FwdState. 그의 임무는 하나였다 — 원본 서버에서 내용을 가져오는 것.

선발대장은 태어나는 순간부터 임무가 주어졌다. 어디로 가야 하는지(목적지 서버), 무엇을 가져와야 하는지(HTTP 요청), 가져온 것을 어디에 넣어야 하는지(StoreEntry). 이 세 가지가 FwdState의 생명이었다.

▼ src/FwdState.h — FwdState 핵심 구조
class FwdState : public RefCountable,
                 public PeerConnectionPopper::Callback
{
public:
    static void Start(const Comm::ConnectionPointer &clientConn,
                      StoreEntry *entry,
                      HttpRequest *request,
                      const AccessLogEntry::Pointer &al);

    Comm::ConnectionPointer clientConn;
    StoreEntry *entry;
    HttpRequest *request;

    // 시도할 목적지 목록
    Comm::ConnectionList serverDestinations;
    Comm::ConnectionPointer serverConn;  // 현재 서버 연결

    time_t start_t;   // 시작 시각
    CachePeer *_peer; // 피어 (직접 연결이면 nullptr)
    int n_tries;      // 시도 횟수

private:
    void start(Pointer aSelf);
    void connectStart();
    void dispatch();   // HTTP 전송 시작
    void complete();   // 응답 완료
    void reforward();  // 실패 시 재시도
};

FwdState는 하나의 HTTP 요청-응답 사이클을 관리하는 상태 기계다. 연결 시도, 실패, 재시도, 완료까지 전체 과정을 조율한다. RefCountable로 비동기 콜백 중 메모리 안전성을 보장한다.

선발대장의 임무는 단계적으로 진행되었다.

Start()
  → peerSelect()           // 목적지 목록 결정 (직접 or 피어)
  → connectStart()         // 첫 번째 목적지로 연결 시도
  → noteConnection()       // 연결 성공/실패 콜백
    → dispatch()           // 성공: HTTP 요청 전송
    → reforward()          // 실패: 다음 목적지 시도
  → complete()             // 응답 수신 완료
  → ~FwdState()            // 임무 완료, 소멸

각 단계는 비동기였다. 연결 시도 후 결과를 기다리는 동안 선발대장은 잠들었다가, 결과가 오면 깨어났다. 이벤트 루프가 그 사이에 다른 수천 개의 일을 처리했다.

제 2 장 FwdState — 임무의 해부

선발대장이 처음 해야 할 일은 목적지를 정하는 것이었다. 이것이 peerSelect()의 역할이었다.

peerSelect()는 복잡한 결정을 내렸다. 직접 원본 서버로 갈 것인가, 피어 캐시를 거칠 것인가? squid.conf의 cache_peer 설정, never_direct, always_direct ACL, ICP/HTCP 응답 등을 종합적으로 고려했다.

peerSelect()는 비동기적으로 동작한다. ICP 쿼리를 보내고 응답을 기다리며, ACL을 확인하고 최종 목적지 목록을 구성한다. 완료 시 콜백으로 FwdState에 알린다.

목적지 목록이 만들어지면 connectStart()가 호출되었다. 목록의 첫 번째 목적지로 TCP 연결을 시작했다.

연결 결과가 오면 noteConnection()이 호출되었다. 성공이면 dispatch()로 HTTP 요청을 전송했다. 실패이면 reforward()로 다음 목적지를 시도했다.

선발대장의 여정이 시작되었다.

제 3 장 DNS 점쟁이에게 묻다

목적지를 정하려면 먼저 주소를 알아야 했다. 도메인 이름이 있었다. www.example.com. 하지만 TCP 연결에는 IP 주소가 필요했다.

DNS 점쟁이에게 물어봐야 했다.

DNS 조회는 src/ipcache.cc와 src/dns/에서 관리되었다. 조회 요청을 하고 비동기로 기다렸다.

▼ src/ipcache.cc — 비동기 DNS 조회
// 비동기 IP 주소 조회
void
ipcache_nbgethostbyname(const char *name,
                         IPH *handler,
                         void *handlerData)
{
    ipcache_entry *i = ipcache_get(name);

    if (i == nullptr) {
        // 캐시에 없음 — 실제 DNS 조회 시작
        i = new ipcache_entry(name);
        ipcache_add(name, i, nullptr, 0);
    }

    if (i->flags.negcached) {
        // 네거티브 캐시 — 존재하지 않는 호스트로 기록됨
        handler(nullptr, handlerData);
        return;
    }

    if (i->addrs.count > 0 && !ipcacheExpiredEntry(i)) {
        // 캐시 히트 — 즉시 콜백
        handler(&i->addrs, handlerData);
        return;
    }

    // 캐시 미스 — DNS 쿼리 대기열에 추가
    i->pending_head = cbdataReference(handlerData);
    idns_query_A(name, ipcacheHandleReply, i);
}

// DNS 응답 도착 시
static void
ipcacheHandleReply(void *data, const rfc1035_rr *answers,
                   int na, const char *error_message)
{
    ipcache_entry *i = (ipcache_entry *)data;

    if (na > 0) {
        // IP 주소들 저장
        // IPv4와 IPv6 모두 처리
        for (int k = 0; k < na; k++) {
            i->addrs.in_addrs[k] = /* IP from answers[k] */;
        }
        i->flags.negcached = 0;
    } else {
        // DNS 실패 — 네거티브 캐시 등록
        i->flags.negcached = 1;
    }

    // 대기 중인 모든 핸들러들에게 알림
    ipcacheCallback(i, 1, squid_curtime);
}

ipcache는 DNS 결과를 캐시한다. 같은 호스트에 대한 여러 요청이 동시에 들어와도 DNS 쿼리는 한 번만 나간다. 결과가 오면 대기 중인 모든 핸들러에게 한꺼번에 알린다. 네거티브 캐시(존재하지 않는 호스트)도 저장하여 반복적인 실패를 방지한다.

DNS 점쟁이는 특별한 능력을 가지고 있었다. 과거의 질문을 기억하는 능력이었다. ipcache — IP 주소 캐시.

www.example.com을 처음 조회하면 DNS 서버에 물어봤다. 답이 오면 ipcache에 저장했다. 다음에 같은 이름을 조회하면 캐시에서 즉시 답했다. DNS 서버까지 갈 필요가 없었다.

캐시에는 TTL이 있었다. DNS 응답에 포함된 TTL값이었다. 예컨대 300초라면 5분 동안은 캐시된 값을 사용했다. TTL이 만료되면 다시 DNS에 물어봤다.

⁂

DNS 조회는 때로 느렸다. 특히 처음 보는 도메인이면 여러 DNS 서버를 거쳐야 할 수도 있었다. 수백 밀리초, 드물게는 수 초까지 걸렸다.

그래서 DNS 조회는 항상 비동기였다. 요청하고 잠들었다가 응답이 오면 깨어났다. 기다리는 동안 이벤트 루프는 쉬지 않았다.

스퀴드의 DNS 클라이언트는 직접 구현되어 있었다. src/dns/ 디렉토리에 있었다. 시스템 라이브러리의 getaddrinfo()를 사용하지 않았다 — 그것이 블로킹 함수이기 때문이었다. 대신 UDP 소켓으로 직접 DNS 서버에 쿼리를 보내고 비동기로 응답을 받았다.

제 4 장 ipcache — 주소부의 기억

ipcache에는 다른 비밀이 있었다. 하나의 도메인이 여러 IP 주소를 가질 수 있었다 — DNS 라운드 로빈이나 GeoDNS. 예컨대 www.google.com은 수십 개의 IP 주소를 반환할 수 있었다.

스퀴드는 이 여러 주소들을 모두 기억했다. 그리고 연결 시 순서대로 시도했다. 첫 번째 IP로 연결이 실패하면 두 번째 IP를 시도했다. 이것이 연결 복원력(connection resilience)의 기초였다.

▼ src/ipcache.cc — 다중 IP 관리
class ipcache_addrs
{
public:
    Ip::Address *in_addrs;  // IP 주소 배열
    unsigned char *bad_mask; // 실패한 IP 마킹

    unsigned char count;     // 총 IP 수
    unsigned char cur;       // 현재 인덱스 (라운드 로빈)

    bool have() const { return count > 0; }

    // 다음 사용할 IP 반환 (라운드 로빈)
    const Ip::Address ¤t() const {
        return in_addrs[cur];
    }

    // 현재 IP가 실패 → 다음으로
    void markBad() {
        bad_mask[cur] = 1;
        next();
    }

    void next() {
        if (++cur >= count) cur = 0;
    }
};

ipcache_addrs는 DNS가 반환한 모든 IP 주소를 보관한다. bad_mask로 실패한 IP를 마킹하여 재시도를 방지한다. 라운드 로빈으로 주소를 순환하여 부하를 분산한다.

IP 주소에는 두 가지 버전이 있었다. IPv4와 IPv6. 현대의 DNS는 둘 다 반환할 수 있었다. A 레코드(IPv4)와 AAAA 레코드(IPv6).

어떤 것을 먼저 시도할 것인가? 이것이 다음 장에서 등장하는 행복한 연결술사의 영역이었다.

하지만 먼저 피어 캐시 이야기를 해야 했다.

제 5 장 피어 캐시 — 이웃 왕국들

왕국은 혼자가 아니었다. 이웃 왕국들이 있었다. 피어 캐시(Cache Peers)였다.

피어 캐시는 두 가지 역할을 했다. 부모(parent)와 형제(sibling)였다.

부모는 모든 요청을 처리할 수 있는 상위 프록시였다. 캐시 미스가 나면 부모에게 물어볼 수 있었다. 부모가 캐시를 가지고 있으면 원본 서버까지 가지 않아도 되었다. 기업 환경에서 지사 프록시가 본사 프록시를 부모로 사용하는 경우가 전형적이었다.

형제는 같은 레벨의 프록시였다. 형제에게 "이거 있어?" 하고 물어볼 수 있었지만, 없으면 형제가 직접 가져오지는 않았다. 형제 캐시들이 서로 물어보며 캐시 히트율을 높이는 구조였다.

▼ squid.conf — 피어 설정
# 부모 프록시 설정
# cache_peer 호스트 타입 HTTP포트 ICP포트 [옵션들]
cache_peer parent.isp.com parent 3128 3130 no-query

# 형제 프록시들
cache_peer sibling1.isp.com sibling 3128 3130
cache_peer sibling2.isp.com sibling 3128 3130

# 특정 도메인은 특정 피어로
cache_peer_access parent.isp.com allow cdn_content
cache_peer_access parent.isp.com deny all

# 직접 연결 제어
never_direct allow all_external   # 항상 부모를 통해
always_direct allow local_content # 내부 콘텐츠는 직접

피어 계층구조는 대규모 ISP나 기업 네트워크에서 캐시를 효율적으로 분산하는 데 사용된다. parent는 캐시 미스를 처리할 책임이 있고, sibling은 캐시 히트만 공유한다.

피어 캐시는 src/neighbors.cc와 src/CachePeer.cc에서 관리되었다. 피어들의 목록을 유지하고, 각 피어의 상태(살아있는지, 응답이 빠른지 등)를 추적했다.

피어의 상태를 어떻게 알았을까? 두 가지 방법이 있었다. 과거 경험(Round Trip Time, 성공/실패 비율)과 ICP/HTCP 쿼리였다.

제 6 장 ICP — 이웃에게 소리치기

ICP(Internet Cache Protocol). 이웃 캐시들에게 "이거 있어?"라고 빠르게 물어보는 프로토콜이었다.

ICP는 UDP 기반이었다. 빠르고 가벼웠다. TCP처럼 연결을 맺을 필요가 없었다. 그냥 패킷을 던지고 응답을 기다렸다.

▼ src/icp_v2.cc — ICP 쿼리 패킷 구조
// ICP v2 패킷 헤더
typedef struct {
    uint8_t  opcode;     // ICP_QUERY, ICP_HIT, ICP_MISS 등
    uint8_t  version;    // 2
    uint16_t length;     // 전체 패킷 길이
    uint32_t reqnum;     // 요청 번호 (응답 매칭용)
    uint32_t flags;      // 플래그
    uint32_t pad;        // 패딩
    uint32_t shostid;    // 발신자 IP (구식, 사용 안 함)
} icp_common_t;

// ICP 쿼리 전송 (형제/부모 프록시들에게)
void
neighborsUdpPing(HttpRequest *request,
                  StoreEntry *entry,
                  IRCB *callback,
                  void *callback_data,
                  int *exprep,
                  int *timeout)
{
    // 모든 ICP 피어들에게 동시에 UDP 패킷 전송
    for each peer in icp_peers:
        icp_send_v2(peer, ICP_QUERY, url);

    // 응답 타임아웃 설정 (기본 2초)
    *timeout = 2000;

    // 응답이 오는 족족 callback 호출
}

ICP 쿼리는 모든 피어들에게 동시에 전송된다. 브로드캐스트 방식이다. 가장 먼저 ICP_HIT로 응답하는 피어에게 요청을 보낸다. ICP_MISS면 무시하거나 다른 피어 시도. 응답이 없으면 타임아웃 후 직접 연결.

ICP 쿼리의 흐름은 이러했다. 선발대장이 peerSelect()를 시작했다. ICP를 지원하는 피어들이 있으면 동시에 ICP 쿼리를 보냈다. 타이머를 설정하고 기다렸다.

응답이 오기 시작했다. ICP_HIT — 이 피어가 원하는 것을 가지고 있다! 이 피어를 목적지 목록의 앞에 넣었다. ICP_MISS — 없다. 무시하거나 다음 피어로. ICP_MISS_NOFETCH — 형제 캐시가 "없고, 직접 가져올 수 없다"는 의미.

타임아웃이 되면 응답한 피어들 중 최선의 선택을 했다. HIT 응답이 있으면 그 피어로. 없으면 부모 피어로. 부모도 없으면 직접 원본 서버로.

⁂

HTCP(Hypertext Cache Protocol)도 있었다. ICP보다 발전된 프로토콜이었다. HTTP 헤더를 포함한 더 풍부한 정보를 주고받을 수 있었다. 하지만 ICP보다 복잡하고 덜 널리 사용되었다.

스퀴드는 둘 다 지원했다. cache_peer ... htcp 옵션으로 HTCP를 사용할 수 있었다.

제 7 장 행복한 연결술사

목적지가 정해졌다. IP 주소도 알았다. 이제 TCP 연결을 맺을 차례였다.

그러나 쉽지 않았다. 현대 인터넷에는 두 가지 세계가 공존했다. IPv4의 세계와 IPv6의 세계. 그리고 하나의 도메인이 두 세계 모두에 주소를 가질 수 있었다.

어느 세계로 먼저 갈 것인가?

이 질문에 답하는 것이 행복한 연결술사 HappyConnOpener였다. RFC 8305 "Happy Eyeballs Version 2"를 구현했다.

▼ src/HappyConnOpener.cc — Happy Eyeballs 알고리즘
// RFC 8305 Happy Eyeballs
// IPv6를 먼저 시도. 250ms 내에 실패하면 IPv4도 병렬 시도.
// 먼저 성공하는 쪽을 사용.

void
HappyConnOpener::start()
{
    // IPv6 주소가 있으면 먼저 시도
    if (destinations_.haveIPv6()) {
        attemptConnecting(candidates_.front()); // IPv6
        // 250ms 타이머 시작
        waitingForOtherPeer_ = true;
        eventAdd("HappyConnOpener", noteGaveBirth,
                 this, 0.25, 1);  // 250ms
    } else {
        // IPv6 없으면 바로 IPv4
        attemptConnecting(candidates_.front());
    }
}

void
HappyConnOpener::noteGaveBirth(void *data)
{
    HappyConnOpener *hco = (HappyConnOpener *)data;
    // 250ms가 지났는데 IPv6 연결 미완료
    // → IPv4도 동시에 시도 시작
    if (!hco->prime || !hco->prime->conn->isOpen()) {
        hco->openFreshConnection(hco->spare);
    }
}

void
HappyConnOpener::noteOpeningDone(HappyConnOpenerAnswer &answer)
{
    if (answer.conn) {
        // 성공한 연결 사용
        // 아직 진행 중인 다른 연결은 취소
        cancelSpare();
        callback_(answer);
    } else {
        // 실패 — 다른 후보 시도
        tryNextDestination();
    }
}

Happy Eyeballs의 핵심: IPv6를 먼저 시도하되 250ms 이내에 완료되지 않으면 IPv4를 병렬로 시도한다. 먼저 성공하는 쪽을 사용하고 나머지는 취소한다. 이로써 IPv6 네트워크 문제 시 사용자 경험 저하를 최소화한다.

Happy Eyeballs의 철학은 낙관주의였다. "둘 다 시도해보고 빠른 놈 쓰자". 비효율적으로 보이지만 실제로는 그렇지 않았다. 연결이 성공하면 다른 시도는 즉시 취소되었다. 유일한 낭비는 250ms의 지연 시간이었다.

하지만 그 250ms가 사용자 경험을 구했다. IPv6 연결이 블랙홀에 빠지는 경우가 있었다 — 연결이 수립되는 것처럼 보이지만 데이터가 흐르지 않는 상황. 이런 경우 순수 IPv6 우선 방식이었다면 타임아웃까지 수십 초를 기다려야 했다. Happy Eyeballs는 250ms 후 IPv4를 병렬로 시작하여 이 문제를 우아하게 해결했다.

제 8 장 IPv6와 IPv4의 경주

IPv6 연결 시도가 시작되었다. 목적지 IP 주소 2001:db8::1의 포트 443으로 TCP SYN 패킷이 날아갔다.

250ms의 타이머가 시작되었다. 째깍째깍.

100ms 후 — IPv6 SYN-ACK가 돌아왔다! 연결이 수립되었다. 타이머는 즉시 취소되었다. IPv4 시도는 시작되지 않았다. Happy Eyeballs가 맡은 일을 마쳤다.

아니면 이렇게 될 수도 있었다. 250ms가 지났는데 IPv6 응답이 없었다. 이제 IPv4 연결도 병렬로 시작되었다. 93.184.216.34의 포트 443으로 TCP SYN 패킷이 날아갔다.

IPv4 SYN-ACK가 먼저 왔다. IPv4 연결을 사용했다. IPv6 시도는 취소되었다.

▼ src/comm/ConnOpener.cc — 실제 TCP 연결 시도
void
Comm::ConnOpener::connect()
{
    // 소켓 생성
    int fd = comm_openex(SOCK_STREAM, IPPROTO_TCP,
                          conn_->remote, COMM_NONBLOCKING,
                          conn_->tos, conn_->nfmark,
                          "http server");

    if (fd < 0) {
        // 소켓 생성 실패
        callback_notifyConnection(Comm::ERR_CONNECT, 0);
        return;
    }

    conn_->fd = fd;

    // 비동기 connect() 호출
    int result = comm_connect_addr(fd, conn_->remote);

    if (result == Comm::INPROGRESS) {
        // 연결 진행 중 — epoll/kqueue에 이벤트 등록
        Comm::SetSelect(fd, COMM_SELECT_WRITE,
                         CommConnectCallbackWrapper, this, 0);
        return;
    }

    // 즉시 성공 (로컬호스트 등)
    callback_notifyConnection(Comm::OK, 0);
}

// epoll이 소켓 쓰기 가능 이벤트 알릴 때 호출
void
Comm::ConnOpener::doneConnecting(int fd, void *data,
                                   Comm::Flag status, int xerrno)
{
    // SO_ERROR 확인 — 실제 연결 성공/실패
    int so_error = 0;
    socklen_t len = sizeof(so_error);
    getsockopt(fd, SOL_SOCKET, SO_ERROR, &so_error, &len);

    if (so_error == 0) {
        callback_notifyConnection(Comm::OK, 0);
    } else {
        callback_notifyConnection(Comm::ERR_CONNECT, so_error);
    }
}

TCP 연결은 비동기다. connect() 시스템 콜이 EINPROGRESS를 반환하면 epoll에 쓰기 이벤트를 등록하고 함수를 종료한다. epoll이 소켓이 준비되었다고 알리면 getsockopt(SO_ERROR)로 실제 성공/실패를 확인한다.

비동기 TCP 연결의 메커니즘은 운영체제의 소켓 API를 기반으로 했다. connect()를 논블로킹 소켓에서 호출하면 즉시 EINPROGRESS를 반환했다. 이는 "연결 시도 중, 나중에 확인하라"는 의미였다.

epoll이 소켓이 쓰기 가능 상태가 되면 이벤트를 알렸다. 이때 getsockopt(SO_ERROR)로 연결이 성공했는지 확인했다. 0이면 성공, 다른 값이면 에러코드였다.

이 메커니즘 덕분에 스퀴드는 수천 개의 연결을 동시에 시도하면서도 이벤트 루프를 막지 않았다.

제 9 장 연결 성공의 순간

TCP 3-way 핸드셰이크가 완료되었다. SYN, SYN-ACK, ACK. 연결이 수립되었다.

Happy Eyeballs가 승리한 연결에 대해 FwdState에 알렸다. noteConnection() 콜백이 호출되었다.

FwdState는 연결된 소켓을 받았다. 이제 HTTP 요청을 보낼 수 있었다. HTTPS라면 먼저 SSL 핸드셰이크가 필요했다.

▼ src/FwdState.cc — noteConnection() 처리
void
FwdState::noteConnection(HappyConnOpenerAnswer &answer)
{
    if (!answer.conn) {
        // 연결 실패 — 다른 목적지 시도
        reforward();
        return;
    }

    serverConn = answer.conn;

    // HTTPS인 경우 SSL 핸드셰이크 시작
    if (request->url.getScheme() == AnyP::PROTO_HTTPS ||
        request->flags.ssl_peek) {
        initiateSSL();
        return;
    }

    // HTTP인 경우 바로 요청 전송
    dispatch();
}

void
FwdState::dispatch()
{
    // 서버와의 통신을 담당할 HttpStateData 생성
    HttpStateData *httpState = new HttpStateData(this);
    httpState->start();  // 요청 전송 시작
}

연결 성공 후 HTTPS면 SSL 핸드셰이크, HTTP면 바로 dispatch(). dispatch()는 HttpStateData를 만들어 실제 HTTP 요청 전송을 위임한다. FwdState는 조율자, HttpStateData는 실제 HTTP 통신 담당자다.

연결 성공의 순간. 선발대장은 잠시 멈추어 생각했다.

나는 이 순간을 위해 달려왔다. DNS에 물었고, 피어들에게 외쳤고, IPv6와 IPv4를 경주시켰고, epoll의 신호를 기다렸다. 그리고 이제 — 연결이 맺어졌다. 이 얇은 선 하나가, 이 소켓 파일 디스크립터 숫자 하나가, 수천 킬로미터 떨어진 서버와 나를 잇는다. 이제 요청을 전달해야 한다. 클라이언트가 원했던 것. "저는 youtube.com의 이 페이지가 보고 싶습니다." 그 원함을 서버에게 전해야 한다. 그것이 내 마지막 역할이다.

제 10 장 HTTP 전령 — 요청 전달

HttpStateData가 나섰다. HTTP 통신의 실무자였다.

HttpStateData는 서버에게 HTTP 요청을 보내고 응답을 받는 일을 담당했다. 이 과정은 생각보다 복잡했다.

프록시는 서버에 요청할 때 Via, X-Forwarded-For 헤더를 추가한다. 피어를 통한 경우 전체 URL(absolute form)을 사용하고, 직접 연결인 경우 경로만(origin form) 사용한다. 응답 데이터는 청크 단위로 StoreEntry에 저장된다.

요청 헤더를 구성할 때 프록시는 몇 가지 헤더를 추가하거나 수정했다.

Via: 1.1 squid.example.com (squid/6.x) — "이 요청은 스퀴드 프록시를 거쳤습니다"는 표시. RFC 7230이 요구하는 헤더였다.

X-Forwarded-For: 192.168.1.105 — 원래 클라이언트의 IP 주소. 원본 서버가 실제 클라이언트 IP를 알 수 있도록 해주는 헤더. forwarded_for on/off 설정으로 제어 가능했다.

Cache-Control: max-age=0 — IMS(If-Modified-Since) 재검증 시 추가.

그리고 일부 헤더는 제거했다. 클라이언트가 보낸 Proxy-Authorization은 서버에 전달하면 안 되었다. 그것은 프록시를 위한 인증 정보였지, 서버를 위한 것이 아니었다.

⁂

응답이 오기 시작했다. HTTP/1.1 200 OK\r\n. 헤더들이 줄줄이 왔다. Content-Type: text/html\r\n. Content-Length: 12345\r\n. 빈 줄 — 헤더 끝. 그리고 본문이 왔다.

HttpStateData는 응답 헤더를 파싱했다. Http1::ResponseParser가 이 역할을 담당했다. 파싱된 헤더가 HttpReply 객체가 되었다.

그리고 데이터를 StoreEntry에 계속 추가했다. 클라이언트들이 그것을 스트리밍으로 받아갔다.

제 11 장 비밀 땅굴 — CONNECT 터널

HTTPS 요청이 일반 HTTP 요청과 다른 점이 있었다. 클라이언트는 먼저 프록시에게 특별한 요청을 보냈다.

CONNECT www.youtube.com:443 HTTP/1.1

CONNECT 메서드. "저와 youtube.com:443 사이에 비밀 터널을 만들어 주세요." 이 요청이 왔을 때 스퀴드는 특별한 모드로 진입했다.

▼ src/tunnel.cc — TunnelStateData 터널 생성
class TunnelStateData : public RefCountable
{
public:
    // 양쪽 연결
    Comm::ConnectionPointer clientConn;  // 클라이언트와의 연결

    // 양방향 버퍼
    struct Connection {

};

}

static void
tunnelConnectedComplete(...)
{

    // "HTTP/1.1 200 Connection established\r\n\r\n"
    const char *response = "HTTP/1.1 200 Connection established\r\n\r\n";
    Comm::Write(tunnelState->clientConn, response, strlen(response), ...);

터널은 매우 단순하다. 양쪽 소켓에서 읽어서 반대쪽에 그대로 쓴다. 내용을 해석하지 않는다. TCP 스트림의 바이트들이 프록시를 통해 투명하게 흐른다. SSL 핸드셰이크, 암호화 데이터, 모든 것이 그대로 통과한다.

터널의 핵심은 투명성이었다. 스퀴드는 내용을 보지 않았다. 클라이언트에서 오는 바이트를 서버에 전달하고, 서버에서 오는 바이트를 클라이언트에 전달했다. 그뿐이었다.

그래서 HTTPS가 동작했다. SSL 핸드셰이크가 이 터널 안에서 진행되었다. 클라이언트와 서버가 암호화 키를 교환했다. 이후 모든 데이터는 암호화되어 흘렀다. 스퀴드는 이 암호화된 데이터를 보았지만 해독할 수 없었다 — 암호화 키를 모르니까.

이것이 일반적인 HTTPS 프록시 터널의 동작이었다.

⁂

터널의 성능은 어떠했을까? 데이터 복사가 두 번 일어났다. 소켓에서 버퍼로, 버퍼에서 다른 소켓으로. 이것이 CPU와 메모리를 사용했다.

더 효율적인 방법도 있었다 — splice() 시스템 콜. 두 파일 디스크립터 사이에서 커널 내부에서 직접 데이터를 이동하는 방법이었다. 사용자 공간으로 데이터를 복사하지 않아도 되었다. 하지만 이것은 일부 플랫폼에서만 사용 가능했고 구현이 더 복잡했다.

대용량 트래픽을 처리하는 환경에서 터널의 성능 최적화는 중요한 과제였다. 스퀴드는 지속적으로 이 부분을 개선해왔다.

제 12 장 변장술사 — SSL 핸드셰이크

SSL 범프가 활성화된 경우는 달랐다. 투명한 터널 대신, 스퀴드는 적극적으로 SSL 핸드셰이크에 참여했다.

클라이언트가 CONNECT www.bank.com:443을 보냈다. 일반적인 터널이었다면 서버에 연결하고 투명하게 중계했을 것이다. 하지만 SSL 범프 설정이 있었다.

변장술사 SSL 모듈이 나섰다. src/ssl/ 디렉토리에서.

// splice인 경우: 터널로 전환
// terminate: 연결 종료

SSL Bump는 세 단계로 진행된다. 각 단계에서 ACL을 평가하여 다음 동작을 결정한다. step1에서 SNI를 보고, step2에서 서버 인증서를 보고, step3에서 최종 결정을 내린다. 단계마다 다른 결정을 내릴 수 있어 세밀한 제어가 가능하다.

SSL 범프의 동작을 단계별로 살펴보자.

step1 (ClientHello 단계): 클라이언트가 TLS 핸드셰이크를 시작하는 ClientHello 메시지를 보냈다. 이 안에 SNI(Server Name Indication)가 있었다 — 클라이언트가 연결하려는 도메인 이름이었다. 스퀴드는 이 SNI를 읽고 ssl_bump ACL을 평가했다.

step2 (서버 인증서 단계): 서버에 실제로 연결하여 서버의 SSL 인증서를 받아보았다. 인증서의 내용(도메인, 유효기간, 발급자)을 확인했다. 두 번째 ssl_bump ACL 평가.

step3 (데이터 단계): 최종 결정. bump라면 위조 인증서를 만들어 클라이언트에게 제시하고 중간자가 되었다. splice라면 터널로 전환하여 투명하게 통과했다.

변장술사의 핵심 기술은 동적 인증서 생성이었다. 실시간으로 인증서를 만들어내는 능력. 서버의 도메인 이름, 유효기간, SAN(Subject Alternative Names)을 복사하여 스퀴드 CA로 서명된 인증서를 즉석에서 만들었다.

제 13 장 재시도의 철학

연결이 실패했다. 서버가 응답하지 않았다. 타임아웃이 발생했다. 503 Service Unavailable이 돌아왔다.

선발대장은 어떻게 해야 할까?

포기하지 않았다. reforward()가 있었다.

▼ src/FwdState.cc — 재시도 로직
void
FwdState::reforward()
{
    // 재시도 가능한지 확인
    if (!checkRetriable()) {
        // 재시도 불가 — 에러 반환
        fail(new ErrorState(ERR_CONNECT_FAIL, ...));
        return;
    }

    // 현재 서버 연결 닫기
    serverConn = nullptr;

    // 다음 목적지 시도
    connectStart();
}

bool
FwdState::checkRetriable()
{
    // 멱등성 없는 요청은 재시도 위험
    // POST, PUT은 기본적으로 재시도 안 함
    if (!request->method.isIdempotent())
        return false;

    // 최대 시도 횟수 초과
    if (n_tries >= Config.forward_max_tries)
        return false;

    // 타임아웃 확인
    if (squid_curtime - start_t > Config.Timeout.forward)
        return false;

    // 이 에러는 재시도해도 의미없는 경우들
    // ERR_TOO_BIG, ERR_INVALID_REQ 등
    if (!HIER_ALLOWRETRY(request->hier.ping.p_rtt))
        return false;

    return true;
}

재시도는 신중하게 결정된다. POST, PUT 같은 비멱등 메서드는 재시도 시 데이터 중복 저장 위험이 있어 기본적으로 재시도하지 않는다. 최대 시도 횟수(forward_max_tries, 기본 10회)와 전체 타임아웃(forward_timeout)도 제한한다.

재시도의 철학은 "조심스러운 낙관주의"였다. 한 번 실패했다고 포기하지는 않되, 무한정 시도하지도 않았다.

재시도에는 규칙이 있었다. GET과 HEAD는 멱등하여 재시도해도 안전했다. POST와 PUT은 서버에 데이터를 제출하는 것이라 재시도하면 같은 데이터가 두 번 제출될 수 있었다 — 온라인 결제에서 이중 결제가 발생할 수 있었다. 그래서 기본적으로 재시도하지 않았다.

그리고 전체 연결 타임아웃이 있었다. connect_timeout은 개별 연결 시도의 타임아웃이었고, forward_timeout은 전체 포워딩 작업의 총 허용 시간이었다. 여러 번 재시도해도 총 시간이 이 한도를 넘으면 포기했다.

⁂

에러 발생 시 클라이언트에게 무엇을 보낼까?

ERR_CONNECT_FAIL — 연결 실패. 502 Bad Gateway.
ERR_READ_TIMEOUT — 읽기 타임아웃. 504 Gateway Timeout.
ERR_DNS_FAIL — DNS 실패. 502 Bad Gateway.
ERR_CANNOT_FORWARD — 모든 경로 실패. 503 Service Unavailable.

각 에러 타입에 맞는 에러 페이지가 클라이언트에게 전달되었다. 2권에서 만났던 에러 페이지 시스템이 여기서 활용되었다.

제 14 장 연결 풀 — 재사용의 미학

연결을 맺는 것은 비용이었다. TCP 3-way 핸드셰이크, SSL/TLS 핸드셰이크, DNS 조회... 모두 시간이 걸렸다.

같은 서버에 다시 요청을 보내야 할 때, 새 연결을 만드는 것보다 기존 연결을 재사용하는 것이 훨씬 빨랐다.

이것이 연결 풀(Connection Pool)의 존재 이유였다. src/PeerPoolMgr.cc와 src/IdleConnList.cc에 있었다.

▼ src/IdleConnList.cc — 유휴 연결 관리
class IdleConnList
{
public:
    // 유휴 연결 추가
    void add(const Comm::ConnectionPointer &conn);

    // 특정 서버의 유휴 연결 꺼내기
    Comm::ConnectionPointer pop(const Comm::ConnectionPointer &key,
                                 bool retriable);

    // 유휴 연결 정리 (타임아웃된 것들 제거)
    void closeN(size_t count);

private:
    typedef std::vector<Comm::ConnectionPointer> ConnectionVector;
    // 서버별 유휴 연결 목록
    std::map<CacheKey, ConnectionVector> indexed;
};

// 사용 완료된 연결을 풀에 반환
void
FwdState::complete()
{
    // 응답이 완전히 수신됨
    // 연결이 재사용 가능한지 확인 (Keep-Alive, HTTP/1.1)
    if (serverConn->isOpen() && flags.keepalive_sent) {
        // 풀에 반환
        fwdPconnPool->push(serverConn, request->url.host());
    }
    // ...
}

유휴 연결 풀은 서버별로 관리된다. HTTP/1.1 Keep-Alive 또는 HTTP/2의 경우 응답 완료 후 연결을 닫지 않고 풀에 반환한다. 다음 요청 시 풀에서 먼저 찾아본다. 없으면 새 연결을 만든다.

연결 풀의 동작은 간단했다. 응답이 완료되면 선발대장은 서버 연결을 닫지 않고 풀에 반환했다. "혹시 누가 같은 서버에 또 요청할 수 있으니 여기 놔두겠습니다."

다음 요청이 같은 서버로 향하면, FwdState는 먼저 풀을 확인했다. 살아있는 연결이 있으면 그것을 사용했다. TCP 핸드셰이크도, SSL 핸드셰이크도, DNS 조회도 없었다. 즉각적이었다.

풀의 연결들에는 타임아웃이 있었다. 너무 오래 유휴 상태로 있으면 서버가 연결을 끊을 수 있었다. 스퀴드는 pconn_timeout 설정값보다 오래된 연결을 먼저 닫았다. 죽은 연결을 재사용하려다가 에러를 경험하는 것보다 나았다.

제 15 장 귀환 — 응답이 돌아오다

선발대장의 임무가 완료되었다.

원본 서버에서 응답이 왔다. 헤더들, 본문 데이터, 마지막 바이트까지. HttpStateData가 StoreEntry에 차곡차곡 채워 넣었다. StoreEntry가 STORE_OK 상태가 되었다.

FwdState의 complete()가 호출되었다.

▼ src/FwdState.cc — complete() 완료 처리
void
FwdState::complete()
{
    // 응답 상태 코드 확인
    Http::StatusCode status =
        entry->mem().baseReply().sline.status();

    // 5xx 에러는 재시도 가능한 경우 재시도
    if (reforwardableStatus(status) && reforward_count < 10) {
        ++reforward_count;
        reforward();
        return;
    }

    // 응답 완료 — 연결 풀에 반환 또는 닫기
    if (serverConn->isOpen()) {
        if (flags.keepalive_sent && checkGoodStats()) {
            fwdPconnPool->push(serverConn,
                                request->url.host());
        }
        serverConn->close();
    }

    // StoreEntry는 이미 clients에게 스트리밍 중
    // 여기서 entry를 완전히 완성 표시
    entry->complete();

    // 로그 엔트리 완성
    al->hier.note(serverConn, request->url.host());

    // 소멸 — 참조 카운트 감소 → RefCountable이 delete
}

// 재시도 가능한 상태 코드들
static bool
reforwardableStatus(const Http::StatusCode s)
{
    return s == Http::scBadGateway            // 502
           || s == Http::scGatewayTimeout     // 504
           || s == Http::scServiceUnavailable // 503
           || s == Http::scInternalServerError; // 500 (일부)
}

complete()에서 5xx 응답은 자동으로 재시도될 수 있다. 연결 풀에 반환하고, StoreEntry를 완성하고, 로그에 계층 정보를 기록한다. FwdState 자체는 참조 카운팅으로 자동 소멸된다.

응답 데이터는 이미 클라이언트에게 스트리밍 중이었다. StoreEntry에 데이터가 들어오는 족족, 기다리던 store_client들이 그 데이터를 받아 클라이언트 소켓으로 내보냈다.

완전히 완성된 StoreEntry. 새로운 요청이 같은 URL을 원하면 선발대장을 부를 필요가 없었다. 창고지기가 바로 꺼내줄 수 있었다. 캐시 히트였다.

⁂

선발대장이 임무를 마치고 돌아섰다. 그는 이미 다음 임무를 준비하고 있었다. 새 캐시 미스가 발생하면 언제든 나설 준비가 되어 있었다.

왕국의 국경 너머로 갔다가 돌아온 이야기가 끝났다. DNS의 신탁을 받아 주소를 알고, 피어들에게 소리쳐 도움을 청하고, 행복하게 연결하고, 때로는 터널을 파고, 때로는 변장하여 SSL 안을 들여다보고, 실패해도 포기하지 않고 다시 시도하고...

그 모든 과정이 클라이언트가 경험하는 수백 밀리초의 지연 안에 담겨 있었다.

이제 마지막 이야기가 남았다. 왕국의 기록관, 성능, 그리고 왕국의 미래.
